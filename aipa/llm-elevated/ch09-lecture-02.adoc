==== Conceptual Core: Function Calling in LLM‑Augmented Systems

The notion of *function calling* (sometimes called *tool use*) extends a language model’s textual competence with deterministic, external capabilities. Rather than asking the model to hallucinate a price, we give it a precise contract—a JSON schema—that describes a callable function. The model’s output is no longer free‑form text but a structured request that an orchestrator can execute.

[quote, "OpenAI Function Calling Documentation", "https://platform.openai.com/docs/guides/function-calling"]
____
A function call is a JSON object that contains the function name and an arguments object that conforms to the function’s parameter schema.
____

The schema for the `search` tool, which we will dissect, looks like this:

[source,json]
----
{
  "name": "search",
  "description": "Run a web search and return the top results.",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {"type": "string"},
      "k": {"type": "integer", "default": 5}
    },
    "required": ["query"]
  }
}
----

*Key Points*

* **Name & Description** – Human‑readable identifiers that the orchestrator and the model use to match intent with capability.
* **Parameters** – A JSON‑Schema object that declares the *shape* of the arguments. It enforces type safety (`string`, `integer`) and can provide defaults (`k` defaults to 5).
* **Required fields** – The model must supply a value for every property listed here; otherwise the call is rejected before execution.
* **Type constraints** – Prevent nonsensical inputs (e.g., a negative integer for `k`) and enable automatic validation in the runtime.

The *orchestrator* (often a thin Python wrapper) receives the model’s function‑call JSON, validates it against the schema, and then forwards the request to the concrete implementation of `search`. This separation of concerns yields several benefits:

1. **Predictability** – The external tool behaves deterministically; the model cannot “make up” results.
2. **Security** – Validation blocks malformed or malicious payloads before they reach the network.
3. **Composability** – Multiple tools (search, summarise, translate) can be chained, each described by its own schema.

The decision‑making process that leads the model to emit a function call is guided by *system prompts* and *function descriptions*. When the model’s internal probability distribution assigns a higher likelihood to a function call than to a plain text continuation, it emits the structured request. In practice, developers tune this behaviour by:

* Providing clear, concise function descriptions that surface the *why* of the tool.
* Supplying examples in the *few‑shot* prompt that demonstrate the desired call pattern.
* Adjusting the `temperature` and `function_call` parameters to bias toward or away from tool use.

**Why does the schema matter?**  
Because it is the *contract* that guarantees the orchestrator can safely translate a model’s intention into concrete action. Without a well‑specified schema, the model could produce ambiguous arguments, leading to failed searches or, worse, unintended network requests.

==== Technical Example: Wiring the `search` Tool in Python

Before we dive into code, imagine what is happening inside the model at the moment it decides to call the search tool. The model has read Maya’s request, recognised the phrase *“cheapest round‑trip flight”* as a domain that changes every day, and therefore inferred that a fresh data source is required. It then consults the function descriptions it was given at start‑up, matches the intent to the `search` description, and constructs a JSON payload that satisfies the schema. This mental choreography is invisible to the user, but it is what we will make explicit in the orchestrator.

Below is a self‑contained example that demonstrates the full lifecycle—from prompt to result—using the OpenAI `gpt‑4‑turbo` model with function calling enabled. The script is deliberately verbose so that each step can be discussed in class.

[source,python]
----
import os, json, openai, httpx
from jsonschema import validate, ValidationError

openai.api_key = os.getenv("OPENAI_API_KEY")

# 1️⃣ Define the function schema exactly as the LLM expects
search_schema = {
    "name": "search",
    "description": "Run a web search and return the top results.",
    "parameters": {
        "type": "object",
        "properties": {
            "query": {"type": "string"},
            "k": {"type": "integer", "default": 5}
        },
        "required": ["query"]
    }
}

# 2️⃣ Helper that performs a real‑world search via a public API (DuckDuckGo Instant Answer)
def real_search(query: str, k: int = 5):
    resp = httpx.get(
        "https://api.duckduckgo.com/",
        params={"q": query, "format": "json", "no_redirect": 1, "skip_disambig": 1},
        timeout=10.0,
    )
    resp.raise_for_status()
    data = resp.json()
    # DuckDuckGo returns a list of “RelatedTopics”; we truncate to k items
    results = [
        {"title": item.get("Text", "No title"),
         "url": item.get("FirstURL", "No URL")}
        for item in data.get("RelatedTopics", [])[:k]
    ]
    return results

# 3️⃣ Orchestrator that validates the JSON against the schema
def orchestrate_function_call(call):
    # Full validation – raises ValidationError on mismatch
    try:
        validate(instance=call["arguments"], schema=search_schema["parameters"])
    except ValidationError as e:
        raise ValueError(f"Invalid arguments: {e.message}")

    args = call["arguments"]
    return real_search(args["query"], args.get("k", 5))

# 4️⃣ Main loop – send a user prompt and let the model decide whether to invoke the tool
def ask_llm(user_input: str):
    response = openai.ChatCompletion.create(
        model="gpt-4-1106-preview",
        messages=[{"role": "user", "content": user_input}],
        functions=[search_schema],
        function_call="auto",  # let the model decide
    )
    choice = response["choices"][0]
    if choice.get("finish_reason") == "function_call":
        # Model wants to call the tool
        call = choice["message"]["function_call"]
        print(f"🔧 Model requested function call: {call['name']}({call['arguments']})")
        try:
            results = orchestrate_function_call(call)
        except Exception as exc:
            # Failure‑mode demo: return a graceful apology
            return f"Sorry, I couldn’t complete the search because: {exc}"
        # Feed the results back into the model for final answer
        follow_up = openai.ChatCompletion.create(
            model="gpt-4-1106-preview",
            messages=[
                {"role": "user", "content": user_input},
                {"role": "assistant", "content": None, "function_call": call},
                {"role": "function", "name": call["name"], "content": json.dumps(results)},
            ],
        )
        final = follow_up["choices"][0]["message"]["content"]
        return final
    else:
        # Model answered directly without a tool
        return choice["message"]["content"]

# 5️⃣ Demo
if __name__ == "__main__":
    query = "latest COVID‑19 case numbers in Brazil"
    answer = ask_llm(query)
    print("\n🗣️ Assistant's answer:\n", answer)
----

*Key Points*

* **Schema‑first definition** – The `search_schema` object mirrors the JSON snippet we examined earlier; any divergence would cause a validation error.
* **Robust validation** – Using `jsonschema.validate` guarantees that required fields are present and typed correctly before the external HTTP request is made.
* **Two‑step interaction** – The model first emits a function call, then receives the raw JSON results as a *function* message, allowing it to produce a polished natural‑language answer.
* **Error handling & graceful degradation** – The `except` block shows how to turn a missing `query` or a rate‑limit (HTTP 429) into a user‑friendly apology rather than a crash.
* **Extensibility** – Adding a new tool (e.g., `summarise`) follows the same pattern: define a schema, implement the function, and let the orchestrator route calls.
* **Failure‑mode mini‑demo** – If the search API returns a 429, `httpx` raises an exception; the orchestrator catches it and returns a helpful message, illustrating how to keep the conversation flowing even when external services misbehave.

Running the script produces an output similar to:

```

🗣️ Assistant's answer:

2. New cases (last 24 h): 12,345
3. Total deaths: 680,123

```

The example demonstrates how a clean JSON contract turns a vague user request into a reliable, reproducible search operation, while also showing how to handle the inevitable hiccups of real‑world APIs.

==== Philosophical Reflection: Trust, Bias, and the Outsourcing of Knowledge

*The incident.* A public‑health chatbot was deployed in a regional hospital to answer patient queries about vaccine schedules. One evening a user asked, “What is the recommended dose of the new flu vaccine for a 70‑year‑old?” The assistant, powered by an LLM with a `search` tool, fetched the top result from a popular health blog that mistakenly listed a pediatric dosage. The chatbot relayed the wrong number, and the patient later reported a mild adverse reaction. The error was traced to three stacked layers of trust that had gone unchecked.

1. **The LLM’s judgment** – The model decided that a fresh web lookup was needed because the query mentioned a “new flu vaccine”. Its internal probability distribution favored a function call, but the prompt engineering did not include a constraint to verify medical sources. Consequently, the model treated any top‑ranked result as authoritative.

2. **The search engine’s ranking algorithm** – The engine’s relevance model gave the blog a high score because the page contained the exact phrase “flu vaccine dose”. Commercial SEO optimisation, not clinical accuracy, drove the ranking. The LLM, blind to these proprietary weights, inherited the bias.

3. **The content on the web** – The internet contains a mixture of peer‑reviewed research, official health agency guidance, and unvetted personal blogs. By returning the first five hits, the `search` tool exposed the assistant to the most visible—not necessarily the most reliable—information.

*What does this mean for epistemic authority?* We move from a monolithic “model knows everything” to a distributed epistemology where knowledge is co‑produced by the model, the tool, and the external information ecosystem. This shift forces designers to ask new questions:

* **Privacy** – The model may embed personally identifying information (e.g., a patient’s name) in the query string. Should the orchestrator scrub or anonymise such data before hitting the web? A common mitigation is to hash or replace identifiable tokens with placeholders before the search call.
* **Bias mitigation** – Schemas can be enriched to require multiple independent sources (`source_count`) or to request a provenance field (`source_url`). The orchestrator can then enforce a rule: *only return an answer if at least two sources agree*.
* **Responsibility** – When a faulty fact is delivered, accountability is diffuse. The model’s developers, the tool’s implementer, and the downstream user all share a portion of the blame. Transparent logging of function calls and results becomes essential for post‑mortem analysis.

*Design patterns for safer outsourcing.*

* **Query sanitisation** – Before the orchestrator forwards a request, run a regex‑based filter that removes email addresses, phone numbers, or medical record numbers.
* **Multi‑source verification** – Extend the schema with an optional `sources` array; after the search, a second tool (`verify`) can cross‑check the claims against a trusted knowledge base.
* **Provenance tracking** – Include a `source` field in every result and surface it to the user (“According to the WHO…”). This not only builds trust but also creates an audit trail.

*Key Points*

* The LLM’s *decision* to call a tool is a probabilistic inference, not a logical proof.
* Search engines embed commercial and algorithmic biases that propagate into the model’s answer.
* Ethical design patterns (query sanitisation, multi‑source verification, provenance tracking) can mitigate but not eliminate these risks.
* Philosophical humility is required: tool‑augmented agents are *assistants*, not ultimate arbiters of truth.
* Designers should treat the schema as a *safety contract* that can enforce provenance, source diversity, and privacy safeguards.

==== Lab Preparation: Implementing the Search Tool in an LLM Agent Framework

*Objective*: By the end of the lab you will have a runnable agent that accepts natural‑language queries, decides whether to invoke the `search` tool, validates the function call against the schema, and returns a polished answer.

*Steps Overview*

1. **Set up the environment** – Install `openai`, `httpx`, and `jsonschema`. Create a virtual environment and export your `OPENAI_API_KEY`.
2. **Copy the schema** – Paste the JSON schema into a `schemas/search.json` file. Use `jsonschema.validate()` in the orchestrator to enforce compliance.
3. **Implement `real_search`** – Choose a free web‑search API (DuckDuckGo, SerpAPI, or a mock endpoint). Write a thin wrapper that respects the `k` parameter and returns a list of `{title, url}` dictionaries.
4. **Build the orchestrator** – Write a function that receives the model’s `function_call` payload, validates it, calls `real_search`, and formats the result as a JSON string.
5. **Create the chat loop** – Use the OpenAI `ChatCompletion` endpoint with `functions=[search_schema]` and `function_call="auto"`. Feed the function result back as a `function` message.
6. **Error handling & logging** – Add try/except blocks around the HTTP request and schema validation. Log each step to a file for later analysis.
7. **Extension challenge** – Add a second tool (`summarise`) and modify the agent to chain `search → summarise`. Update the prompt to instruct the model to use both tools when appropriate.

*Key Points for the Lab*

* The JSON schema is the single source of truth; any change must be reflected in both the LLM request and the validation code.
* Validation prevents malformed calls from reaching the external API, protecting both the system and third‑party services.
* Logging the raw function calls and responses creates an audit trail useful for debugging bias or privacy issues.
* Chaining tools demonstrates how complex tasks (e.g., “Give me a brief report on the latest AI conferences”) can be decomposed into reusable primitives.
* Adding a `source` field to the schema will let you surface provenance in the final answer—a concrete illustration of the philosophical safeguards discussed earlier.
