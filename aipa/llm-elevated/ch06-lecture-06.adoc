=== 6.1: Caching LLM Calls – Why It Matters

[.epigraph]
____
*“Your demo just crashed because the API hit its daily quota. The audience is staring, the investors are waiting, and you have five seconds to keep the conversation going.”*  
How do you guarantee sub‑second responses while keeping the cost of a large‑language‑model (LLM) service under control?
____

==== Conceptual Core

When a live demo stalls, the audience hears a long, awkward silence; investors begin to doubt the viability of the product; and the engineering team scrambles to reboot a quota‑exhausted API. That single moment crystallises the two‑fold problem that every production‑grade LLM system must solve: **latency** (how long the user waits) and **cost** (how much the service bill grows). In practice the latency comes from three sources that stack on every request:

1. a network hop to the cloud provider,  
2. an authentication handshake, and  
3. the model’s own inference pipeline, which—even on a high‑end GPU—adds several hundred milliseconds per token.

If the same prompt is issued repeatedly—think of a FAQ‑style chatbot, a code‑completion assistant, or a classroom demo—the per‑token cost multiplies, and the user experience degrades from “instant answer” to “waiting for a response.” The tension between speed and spend is the engine that drives today’s caching research.

**Caching** is the oldest, most reliable technique for breaking that latency‑cost loop. By storing the *output* of an LLM for a given *input* we can serve future identical requests from memory, eliminating the network round‑trip entirely. However, caching LLM calls is not a naïve “store‑and‑retrieve” problem; it must satisfy three design goals that keep the cache both useful and safe:

* **Deterministic key generation** – The cache must map *exactly* the same prompt (including model name and any generation parameters) to a single, collision‑resistant identifier.  
* **Freshness control** – Model outputs evolve as the underlying model is updated or as the knowledge base changes; a time‑to‑live (TTL) policy prevents serving stale answers indefinitely.  
* **Bounded resource usage** – Memory is finite; an eviction policy (most commonly Least‑Recently‑Used, LRU) guarantees that the cache never exceeds a pre‑defined size.

These goals raise a second set of concrete questions that we will answer step‑by‑step:

* How do we turn a complex prompt‑parameter tuple into a compact identifier that never collides?  
* Which data structure gives us O(1) lookup while also tracking usage order for LRU eviction?  
* How can we enforce TTL without a dedicated cleanup thread that would add its own latency?

The answer is a small, self‑contained Python class that leverages the standard library: `hashlib` for deterministic keys, `OrderedDict` for LRU ordering, and timestamps for lazy TTL checks. The class is deliberately simple so that the core ideas remain visible, yet it is production‑ready: it respects a maximum size, evicts the least‑recently used entry, and automatically discards entries that have outlived their TTL.

**Key points distilled from the core**

1. *Deterministic hashing* turns arbitrarily long prompts into fixed‑size keys, enabling constant‑time dictionary lookups.  
2. *Lazy TTL* checks keep the cache simple—no background thread is required; freshness is verified only on access.  
3. *OrderedDict* supplies both O(1) lookup and LRU ordering; moving a key to the end on each hit updates its “recently used” status.  
4. *Explicit eviction* before storing a new entry guarantees that the cache never exceeds its configured capacity.  
5. *Separation of concerns*—`get` handles retrieval and freshness, `set` handles capacity and insertion—makes the class easy to extend (e.g., thread‑safety, remote back‑ends).  
6. *Cost‑latency trade‑off* can be quantified: a 700 ms API call becomes <10 ms when served from cache, saving both time and dollars.

==== Technical Example

The following implementation of `LLMCache` follows the design goals introduced above. It is deliberately lightweight, using only the Python standard library, but it is structured so that swapping the in‑memory store for a Redis hash or adding async support requires only a few lines of change.

[source,python]
----
import time
import hashlib
from collections import OrderedDict

class LLMCache:
    """
    A lightweight in‑memory cache for LLM responses.
    • ttl_seconds – how long a cached entry stays fresh.
    • maxsize     – maximum number of entries; LRU eviction when exceeded.
    """
    def __init__(self, ttl_seconds: int = 86_400, maxsize: int = 10_000):
        self.ttl = ttl_seconds                     # Freshness control
        self.maxsize = maxsize                     # Resource bound
        self.store = OrderedDict()                 # LRU‑aware container

    def _make_key(self, prompt: str, model: str, **params) -> str:
        """
        Deterministic, collision‑resistant key.
        • Concatenates model, prompt, and sorted parameter items.
        • SHA‑256 yields a fixed‑length, uniform identifier.
        """
        key_material = f"{model}|{prompt}|{sorted(params.items())}"
        return hashlib.sha256(key_material.encode()).hexdigest()

    def get(self, prompt: str, model: str, **params):
        """
        Lookup path (see Figure 1).
        Returns cached response if present *and* fresh; otherwise None.
        """
        key = self._make_key(prompt, model, **params)
        entry = self.store.get(key)

        # Hit & fresh?
        if entry and (time.time() - entry['ts'] < self.ttl):
            # Update usage order for LRU
            self.store.move_to_end(key)
            return entry['response']

        # Stale entry – remove it so it never blocks a fresh result
        if entry:
            del self.store[key]
        return None

    def set(self, prompt: str, model: str, response: str, **params):
        """
        Store a fresh response.
        • Evicts LRU entry if capacity would be exceeded.
        • Inserts new entry at the tail (most‑recent position).
        """
        key = self._make_key(prompt, model, **params)

        # Enforce maxsize via LRU eviction
        if len(self.store) >= self.maxsize:
            # popitem(last=False) removes the oldest (least‑recently used) entry
            self.store.popitem(last=False)

        self.store[key] = {'response': response, 'ts': time.time()}
----
*Figure 2: Full `LLMCache` implementation.*

**Illustrated key points**

* **Hash‑based keys** guarantee O(1) lookup regardless of prompt length.  
* **Lazy TTL** discards stale entries only when they are accessed, avoiding extra threads.  
* **OrderedDict** implements LRU with `move_to_end` and `popitem(last=False)`.  
* **Separation of concerns** keeps `get` focused on retrieval and freshness, `set` on capacity.  
* **Extensibility** – swapping `self.store` for a Redis client or wrapping the methods with `asyncio.Lock` requires minimal code changes.

To see the cache in action, run the short script below. It simulates an LLM call with a `time.sleep` delay, demonstrates a cache hit on the second request, and forces an eviction by exceeding `maxsize`.

[source,python]
----
import time

def mock_llm(prompt):
    """Simulate latency of a real LLM call."""
    time.sleep(0.7)               # 700 ms artificial delay
    return f"Response to: {prompt}"

cache = LLMCache(ttl_seconds=30, maxsize=3)

prompt = "Explain Newton's first law."
# First request – miss, calls mock LLM
resp = cache.get(prompt, model="gpt‑4")
if resp is None:
    resp = mock_llm(prompt)
    cache.set(prompt, model="gpt‑4", response=resp)
print("First:", resp)

# Second request – hit, returns instantly
resp = cache.get(prompt, model="gpt‑4")
print("Second (cached):", resp)

# Fill cache to trigger eviction
for i in range(4):
    p = f"Prompt {i}"
    cache.set(p, model="gpt‑4", response=mock_llm(p))
print("Cache keys after overflow:", list(cache.store.keys()))
----
Running this script will show a ~700 ms pause on the first call, an instantaneous second call, and finally the eviction of the oldest entry once the size limit is breached.

==== Philosophical Reflection

Caching is often presented as a purely engineering optimisation, yet it raises profound questions about the *nature of knowledge* in AI‑augmented systems. When we serve a cached answer, we are **reifying a snapshot of a model’s belief** at a particular moment in time. If the underlying model is updated—perhaps because new data has been incorporated or a safety patch applied—our cache may continue to propagate an outdated worldview.

*Case study: a medical chatbot.* Imagine a virtual health assistant that answers “What is the recommended dosage of Drug X?” The LLM was trained on guidelines from 2022. In early 2024 the FDA issues a revised dosage limit. A naïve cache with a 30‑day TTL will still return the 2022 recommendation for weeks after the change, potentially exposing patients to harmful overdoses. The ethical responsibility therefore lies not only in implementing TTLs but also in **designing governance policies** that dictate how often caches must be invalidated after critical model revisions.

This tension mirrors classic epistemological debates: the trade‑off between *stability* (repeatable, reliable answers) and *freshness* (up‑to‑date, possibly more accurate answers). In the AI context, the stakes are amplified because a cached answer can be delivered to millions of users in milliseconds. The following table illustrates a realistic cost‑latency trade‑off for a chatbot handling 10 000 daily queries:

[options="header"]
|=== 
|Metric |Without cache |With cache (TTL = 1 hour) 
|Average latency |0.68 s (network + inference) |0.012 s (in‑process lookup) 
|API cost (USD) |$12.50 (≈ $0.00125 per call) |$0.30 (≈ $0.00003 per hit) 
|Stale‑answer risk |0 % (always fresh) |≈ 2 % (entries older than 1 h) 
|===

The table makes explicit that a modest TTL can slash both latency and cost by two orders of magnitude, but it also quantifies the residual risk of serving stale information. Designers must therefore **encode a value judgment** in the TTL setting: how much staleness is acceptable for the domain at hand?

*Take‑away reflections*

1. **Temporal fidelity** – TTL is a technical knob, but its setting encodes a policy about how quickly knowledge must be refreshed.  
2. **Responsibility diffusion** – Caches shift part of the correctness burden from the model provider to the system integrator, who must monitor model updates and regulatory changes.  
3. **Bias amplification** – Repeatedly serving the same cached answer can cement systematic biases present in the original model snapshot.  
4. **Transparency** – Users should be informed when a response originates from cache versus a fresh inference; this supports trust and allows users to request an “up‑to‑date” answer if needed.  
5. **Governance** – Organizational policies must define cache‑invalidation triggers (e.g., model version bump, legal amendment) and audit trails for cached responses.  

By foregrounding these philosophical dimensions, we move beyond “speed hacks” to a more responsible, human‑centered approach to AI system design.
