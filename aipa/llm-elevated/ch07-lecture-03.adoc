=== 7.3: RAG — Retrieval‑Augmented Generation
:leveloffset: 2

[.epigraph]
"Retrieval grounds generation—or tries to."
— Lewis et al., "Retrieval‑Augmented Generation" <<Lewis2020>>

*Hook*: A customer‑service chatbot once answered a legal question with a fabricated clause, “the contract may be terminated with 30 days’ notice,” even though no such clause existed in the company’s policy. After wiring a retrieval step that pulled the actual policy document, the bot began citing the correct paragraph and stopped hallucinating. *Retrieval can turn a confident liar into a trustworthy assistant.*

==== Example Prompts

* "Answer based on our company docs." — _Retrieve → inject into prompt → generate_
* "Summarize the relevant passages." — _RAG pipeline: query → embed → search → augment → generate_
* "Cite where you found that." — _Retrieved chunks as sources; grounding in retrieval_

==== Conceptual Core

**The problem** – Large language models excel at fluent generation but frequently fabricate details that are not present in any source. In high‑stakes domains (legal, medical, finance) such hallucinations can be costly or dangerous.  

**The solution** – Retrieval‑Augmented Generation (RAG) inserts factual context from an external knowledge store directly into the prompt, giving the model a concrete grounding point. By augmenting the query with retrieved passages, we bias the generation toward verifiable information without retraining the model.  

**The limitation** – Retrieval is only as good as the index it draws from. Garbage‑in‑garbage‑out still applies: irrelevant or incorrect chunks can mislead the model, and the model may still ignore or distort the retrieved text.  

[.formal-definition]
****
*Formal Definition (RAG)*
*RAG* (Retrieval‑Augmented Generation) is a pipeline in which retrieval augments the prompt before generation. Given a query *q*, RAG: (1) retrieves top‑*k* chunks from a vector store via similarity search, (2) constructs an augmented prompt = [retrieved context] + [q], (3) generates via the LLM. Knowledge is added by retrieval, not by retraining. See <<Lewis2020>>.
****

**Latency vs. quality trade‑off** – Choosing *k* and whether to rerank affects both response time and answer fidelity. Larger *k* improves recall but adds latency; reranking (e.g., with a cross‑encoder) sharpens precision at the cost of extra compute. Practitioners often adopt a two‑stage approach: retrieve a generous set, then filter to a concise, high‑quality subset before prompting.  

**Mini‑case study** – Consider a SaaS support bot that answers “How do I reset my password?” The baseline model replies, “You can reset it from the account settings page,” which is correct but unsourced. After integrating RAG with the product’s knowledge base, the bot returns: “According to the *User Guide* (Section 3.2), click *Forgot password* on the login screen, then follow the emailed link.” The answer is both accurate and explicitly cited, demonstrating how retrieval reduces hallucination and improves auditability.

===== Key Points

* **Identify** the hallucination problem and why grounding matters  
* **Add** factual context by retrieving top‑*k* chunks before generation  
* **Understand** that retrieval quality limits overall performance  
* **Balance** latency and precision via *k* selection and optional reranking  
* **Choose** chunking strategies: fixed, semantic, sliding‑window, hierarchical  
* **Apply** RAG to factual QA, domain‑specific tasks, citation, and hallucination reduction  
* **Recognize** that retrieval is a strong hint, not a guarantee of truth  
* **Measure** accuracy, citation fidelity, and faithfulness  

==== Technical Example

[.algorithm]
****
*RAG Pipeline*  
.Input: Query *q*, vector store *V*, LLM *M*, *k*  
.Output: Generated answer  
+
1. embed(*q*) → *q_emb*  
2. *V*.search(*q_emb*, *k*) → *chunks*  
3. (optional) rerank *chunks* → *top_chunks*  
4. prompt = "Context: [top_chunks]\nQuery: [q]\nAnswer based on the context."  
5. *M*.generate(prompt) → *answer*  
6. Return *answer*  
****

**Implementation checklist** – Embed the query, retrieve *k* candidate chunks, optionally rerank with a cross‑encoder or a secondary LLM call, concatenate the selected passages into a prompt template, and invoke the generation model.  

**Evaluation strategy** – Run the pipeline on a held‑out factual QA set both with and without retrieval. Compare (i) answer accuracy, (ii) citation fidelity (does the model correctly reference the retrieved chunk), and (iii) faithfulness (does the answer stay within the retrieved context).  

===== Key Points

* **Implement** embed, retrieve, optional rerank, prompt, generate  
* **Use** a concise prompt template that merges context and query  
* **Evaluate** with vs. without retrieval on factual QA  
* **Optionally** add reranking to boost precision at the cost of latency  
* **Report** accuracy, citation fidelity, and faithfulness  

==== Philosophical Reflection

Grounding in retrieval: we hope retrieval grounds generation. We provide the model with a source; we expect it to use it. The boundary between *retrieved* and *generated* is blurry. The model may quote verbatim, paraphrase, synthesize, or invent. It may cite a passage and add fabrication. It may ignore the context entirely. Retrieval reduces hallucination on average—but it does not eliminate it. The model is still a generator; we are biasing it toward the retrieved content, not constraining it.

**What does “grounded” mean?** – One definition: the answer is entailed by or supported by the retrieved context. Another: the model explicitly cites its sources. A third: the answer does not contradict the retrieved material. These notions diverge, so evaluation must state which criterion is being measured. Citation fidelity—does the model attribute correctly?—is distinct from factual accuracy. A model can cite accurately and still be wrong; it can be right and fail to cite. Both matter for trustworthy RAG.

**Epistemic authority** – By outsourcing knowledge to a vector store, the system shifts the locus of truth from the model’s parameters to the curated index. Who curates the store? Who decides what gets retrieved? RAG encodes epistemic authority in the retrieval pipeline, making the store a gatekeeper of what the model can “know.”  

===== Key Points

* **Acknowledge** that grounding is a hope, not a guarantee  
* **Recognize** the blurry line between retrieved and generated text  
* **Define** “grounded” explicitly for each evaluation scenario  
* **Separate** citation fidelity from factual accuracy  
* **Understand** that the vector store becomes the epistemic authority
