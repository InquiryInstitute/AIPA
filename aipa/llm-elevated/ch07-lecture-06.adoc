=== 7.6: Memory‑Augmented Conversational Agents

[.epigraph]
____
*“Imagine you are chatting with an AI that suddenly forgets what you asked five minutes ago. How would you design a system that remembers?”*

==== Hook

Picture a busy e‑commerce site during a holiday sale. A customer, Maya, opens a chat window and asks, “Where is my order #12345?” The bot, built on a vanilla large language model, replies with a generic apology because it only sees the last utterance and has no notion of Maya’s prior order‑history query. In the real world this would spark frustration, a support ticket, and possibly a lost sale. The tension that drives today’s lecture is **how to give a conversational LLM a reliable, searchable memory so that every response is anchored in what has already been said and in external facts that the model itself does not store**. We will unpack a concrete five‑step pipeline, see it in action, and then step back to ask what it means for a machine to “remember”.

==== Conceptual Core

Memory for LLM‑driven agents can be understood as a *pipeline* that repeats five tightly coupled operations on every turn. Throughout this section we will follow a running example: a user asks the bot to “change the shipping address for order #12345”. Each operation is motivated by a concrete problem that would arise if the step were omitted.

1. **Capture the user query.**  
   The raw utterance arrives from the front‑end, but before it can be processed we must *sanitize* (remove PII, escape special characters) and *enrich* it with metadata such as a timestamp, a unique user identifier, and the communication channel (web, voice, SMS). Without this enrichment the downstream retriever cannot apply recency weighting or enforce access controls, and the system becomes vulnerable to injection attacks.

2. **Retrieve relevant context.**  
   A vector‑store (FAISS, Elastic, Pinecone, …) is queried with an embedding of the sanitized query. The store returns the *k* most similar snippets, which may be whole documents, paragraph‑level excerpts, or previously generated responses. The quality of these snippets hinges on two factors: (a) the fidelity of the embedding model (semantic similarity vs. lexical overlap) and (b) the granularity of the index (storing whole orders versus line‑by‑line transaction logs). If the retriever pulls an outdated shipping address, the bot will propagate the wrong information.

3. **Compose a prompt and generate an answer.**  
   The retrieved snippets are concatenated with a system instruction (e.g., “You are a helpful customer‑service assistant”) and the user query to form a *context‑aware* prompt. Because LLMs have a hard token limit, we must decide how much of each snippet to keep, possibly summarising or truncating. The LLM then produces a response that is *grounded* in the retrieved evidence rather than hallucinating from its internal weights alone.

4. **Optionally invoke external tools.**  
   If the LLM detects a gap—say it needs the current shipping status from an order‑management API—it can emit a structured tool‑call (often a JSON payload). The orchestrator executes the call, receives the result, and injects it back into the prompt for a second‑stage generation. This step lets the system extend beyond static corpora to live, authoritative data.

5. **Persist the interaction.**  
   The full turn—query, retrieved context, generated answer, any tool results, and the metadata—gets stored in a *memory buffer* (commonly a relational table). Future retrievals treat these stored episodes as part of the corpus, closing the feedback loop. Persisted episodes must be timestamped, tagged (e.g., “order‑update”), and, where required, encrypted or set to expire to satisfy privacy regulations.

*Key Points – Conceptual Core*

* Memory is **episodic**: each turn is a discrete, indexable episode.  
* Retrieval quality depends on **embedding fidelity** and **index granularity**.  
* The LLM’s **context window** imposes a hard ceiling on how much retrieved text can be injected.  
* **Tool calls** enable dynamic grounding beyond static knowledge bases.  
* Persisted episodes must be **timestamped**, **tagged**, and **encrypted** for privacy compliance.  
* **Recency‑weighted retrieval** helps the agent focus on the most recent, relevant facts.  
* **Bias amplification** can occur when early mistakes are stored and later retrieved.  
* **Token budgeting** (splitting, summarising) is essential to stay within model limits.  

==== Technical Example

Below is a fully annotated Python script that implements the five‑step pipeline. The surrounding prose explains *why* each block exists, not merely *what* it does, so you can follow the logical flow before diving into the code.

[source,python]
----
import os
import sqlite3
from datetime import datetime
import openai               # LLM client
import faiss                # Vector store
import numpy as np

# ----------------------------------------------------------------------
# 1️⃣ Capture user input
# ----------------------------------------------------------------------
def get_user_query():
    """Read a line from stdin, sanitise it, and attach metadata."""
    query = input("🗣️  You: ")
    # Simple sanitisation – strip leading/trailing whitespace
    clean_query = query.strip()
    return {
        "text": clean_query,
        "timestamp": datetime.utcnow().isoformat(),
        "user_id": os.getenv("USER_ID", "anonymous")
    }

# ----------------------------------------------------------------------
# 2️⃣ Retrieve relevant context
# ----------------------------------------------------------------------
def retrieve_snippets(query_text, k=4):
    """Embed the query, search FAISS, and return top‑k snippets."""
    # Assume a deterministic embedding function is available
    query_vec = embed_text(query_text)               # → np.ndarray
    distances, indices = faiss_index.search(query_vec, k)
    # Guard against empty results

        return []

    return snippets

# ----------------------------------------------------------------------
# 3️⃣ Compose prompt & generate answer
# ----------------------------------------------------------------------
def generate_answer(query, snippets):
    """Build a prompt that includes system instructions and retrieved snippets."""
    system_prompt = "You are a helpful customer‑service assistant."
    # Separate snippets clearly; truncate if token budget is exceeded
    context = "\n---\n".join(snippets)
    full_prompt = f"{system_prompt}\n\nContext:\n{context}\n\nUser: {query}"
    response = openai.ChatCompletion.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": full_prompt}]
    )
    return response["choices"][0]["message"]["content"]

# ----------------------------------------------------------------------
# 4️⃣ Optional tool call (weather API example)
# ----------------------------------------------------------------------
def maybe_call_tool(answer):
    """Detect a placeholder token and call an external API."""
    if "<weather>" in answer:
        # Real implementation would perform an HTTP request here
        weather = "Sunny, 22°C"
        return answer.replace("<weather>", weather)
    return answer

# ----------------------------------------------------------------------
# 5️⃣ Persist interaction
# ----------------------------------------------------------------------
def store_interaction(conn, query, snippets, answer, tool_result=None):
    """Insert a row into the memory table for later retrieval."""
    cur = conn.cursor()
    cur.execute(
        """
        INSERT INTO memory (user_id, timestamp, query, context, answer, tool_result)
        VALUES (?, ?, ?, ?, ?, ?)
        """,
        (
            query["user_id"],
            query["timestamp"],
            query["text"],
            " || ".join(snippets),      # simple delimiter for storage
            answer,
            tool_result
        )
    )
    conn.commit()

# ----------------------------------------------------------------------
# Main loop
# ----------------------------------------------------------------------
if __name__ == "__main__":
    # Initialise SQLite memory DB
    conn = sqlite3.connect("memory.db")
    conn.execute("""CREATE TABLE IF NOT EXISTS memory (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        user_id TEXT,
        timestamp TEXT,
        query TEXT,
        context TEXT,
        answer TEXT,
        tool_result TEXT
    )""")
    # Assume FAISS index and embedding function are already loaded
    while True:
        user = get_user_query()
        snippets = retrieve_snippets(user["text"])
        raw_answer = generate_answer(user["text"], snippets)
        final_answer = maybe_call_tool(raw_answer)
        print(f"🤖  Bot: {final_answer}")
        store_interaction(conn, user, snippets, final_answer)
----
*Key Points – Technical Example*

* **Sanitise** user input before embedding to avoid injection attacks.  
* **Deterministic embeddings** keep retrieval stable across runs.  
* **FAISS index** must be refreshed (re‑indexed) whenever new episodes are stored.  
* **Prompt construction** respects the LLM’s token limit; truncate or summarise snippets when necessary.  
* **Tool‑call detection** can be a simple placeholder or a structured JSON schema; the latter enables richer orchestration.  
* **Memory persistence** in a relational table enables ad‑hoc SQL queries (e.g., “last 5 turns for user X”).  
* **Error handling** for empty retrieval results prevents the LLM from receiving an empty context.  
* **Logging** of token counts and latency helps diagnose performance bottlenecks.

The script above mirrors the conceptual pipeline word‑for‑word: `get_user_query` implements **Capture**, `retrieve_snippets` performs **Retrieve**, `generate_answer` carries out **Compose & Generate**, `maybe_call_tool` embodies **Optional Tool Call**, and `store_interaction` executes **Persist**. By keeping the code structure aligned with the high‑level steps, we make the system both transparent to developers and auditable for compliance teams.

==== Philosophical Reflection

When a machine can retrieve past interactions and weave them into new answers, we are forced to ask **what counts as “knowledge” for an artificial system?** Retrieval‑augmented generation gives the illusion of memory, yet the underlying LLM remains a statistical predictor that has never *experienced* the retrieved facts. The line between *remembered evidence* (the snippets) and *generated inference* (the LLM’s prose) mirrors the human tension between episodic recollection and abstract reasoning. Does a bot that says, “Your order shipped yesterday” because it fetched a live API response *know* the shipping status, or is it merely *pretending* to know by stitching together a fresh data point?

Beyond epistemology, the persistent storage of user utterances raises **ethical obligations**. Users may invoke a “right to be forgotten”, but a naïve memory buffer can become a de‑facto surveillance log. Designers must therefore embed data‑minimisation, encryption at rest, and automatic expiration policies. Moreover, the feedback loop—where stored episodes influence future retrieval—can amplify early mistakes. If the bot once mis‑identified an order number and stored that error, subsequent turns may repeatedly surface the same misinformation, a phenomenon akin to “confirmation bias” in human memory. Recognising these limits prepares us for the next lecture, where we will explore **hierarchical knowledge graphs** that compress episodic traces into structured, long‑term representations, reducing both storage overhead and bias propagation.

*Key Points – Philosophical Reflection*

* Retrieval provides *grounded evidence*; generation supplies *interpretation*.  
* The system “knows” only insofar as it can reliably retrieve and correctly integrate external data.  
* Persistent storage creates a **privacy tension**: utility vs. the right‑to‑be‑forgotten.  
* Self‑referential memory loops can **reinforce bias** and propagate errors.  
* Ethical design demands **encryption**, **access controls**, and **data‑expiry** policies.  
* Future architectures will shift from raw episodic buffers to **knowledge graphs** for scalable, trustworthy long‑term memory.
