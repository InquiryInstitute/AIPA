==== Conceptual Core

*Hook – a story of a spam filter*  
Imagine you are building a spam filter that classifies 10 000 emails. On the training set it achieves **100 % accuracy**—every single message is correctly labelled. Yet, when you deploy the model to a fresh inbox, it misclassifies half of the new messages. The culprit is **over‑fitting**: the model has memorised idiosyncrasies of the training data (typos, specific header ordering) that do not generalise. The central question becomes: **how can we force the learner to ignore noise and capture only the underlying signal?**

*Regularization as a penalty on complexity*  
The answer lies in adding a **penalty term** to the loss function. By augmenting the empirical risk  

```
L̂(w) = 1/n Σℓ(y_i, f_w(x_i)) + Ω(w)
```  

we ask the optimiser to prefer weight vectors that are “small” or “sparse” in a quantifiable sense. Three families dominate practice:

* **L1 (Lasso)** – `Ω(w)=λ∑|w_i|`. The absolute‑value penalty drives many coefficients exactly to zero, yielding an automatically sparse model.
* **L2 (Ridge)** – `Ω(w)=λ∑w_i²`. Squaring each weight shrinks them toward zero but rarely eliminates them, preserving every feature with reduced magnitude.
* **Elastic Net** – `Ω(w)=α·λ∑|w_i| + (1‑α)·λ∑w_i²`. By mixing the two, we can balance sparsity against uniform shrinkage.

*Bias‑variance trade‑off visualised*  
From a statistical viewpoint, regularisation implements a **bias‑variance trade‑off**. As λ grows, **bias** increases (the model is forced to be simpler) while **variance** falls (the model becomes less sensitive to fluctuations in the training set). The classic U‑shaped curve of validation error versus λ illustrates this: a small λ under‑regularises (high variance), a very large λ over‑regularises (high bias), and an intermediate λ minimises out‑of‑sample error.

*Cross‑validation as a data‑driven compass*  
Choosing the “right” λ (and the regulariser type) requires a principled search. **k‑fold cross‑validation** splits the data into *k* equally sized folds, iteratively training on *k‑1* folds and validating on the held‑out fold. The mean validation metric across folds provides a low‑variance estimate of generalisation performance. Variants such as **leave‑one‑out** (k = n) or **nested CV** (an outer loop for model selection, an inner loop for hyper‑parameter tuning) further guard against optimistic bias when the validation set is small.

*Model selection workflow*  
Putting the pieces together yields a systematic pipeline:

1. Define a **grid** (or random/Bayesian search space) over regularisation strength λ, mixing parameter α (for Elastic Net), and regulariser type (L1, L2, none).  
2. For each candidate, run **k‑fold CV** and record the mean validation score.  
3. Select the candidate with the highest score, then **re‑train on the full training set** using the chosen hyper‑parameters.  

The `ml_trainer` API mirrors this workflow: it accepts a regulariser specification, a λ value, and a `cv_folds` argument, runs the CV loop internally, and returns the best‑scoring model together with the selected hyper‑parameters.

===== Key Points

* Over‑fitting manifests as perfect training accuracy but poor test performance.  
* Regularisation adds a penalty `Ω(w)` that encourages simplicity.  
* L1 → sparsity (exact zeros); L2 → uniform shrinkage; Elastic Net → blend.  
* Bias‑variance trade‑off: λ ↑ ⇒ bias ↑, variance ↓; optimal λ balances the two.  
* k‑fold CV estimates out‑of‑sample error; nested CV mitigates selection bias.  
* Hyper‑parameter grid (λ, α, regulariser) + CV → systematic model selection.  
* `ml_trainer` encapsulates: `regularizer`, `lambda`, `alpha`, `cv_folds`.  

==== Technical Example

We now walk through a **live‑coding notebook** that demonstrates the full pipeline on a synthetic regression problem. First we generate data with 20 features, of which only 5 are truly predictive. Then we explore a log‑spaced λ grid, visualise the **coefficient paths**, and watch the validation score curve shape the final choice.

[source,python]
----
from ml_trainer import Trainer
import numpy as np
import matplotlib.pyplot as plt

# synthetic data: 200 samples, 20 features, 5 informative
X, y = make_regression(n_samples=200, n_features=20,
                       n_informative=5, noise=0.5, random_state=42)

lambdas = np.logspace(-3, 1, 10)          # 0.001 … 10
best_score = -np.inf
best_cfg = None
coeff_paths = []

for lam in lambdas:
    trainer = Trainer(
        regularizer='l1',                 # L1 (Lasso) for sparsity
        lambda_=lam,
        cv_folds=5
    )
    cv_score, coef = trainer.fit(X, y)
    coeff_paths.append(coef)
    print(f'λ={lam:.3f} → CV R²={cv_score:.4f}')
    if cv_score > best_score:
        best_score, best_cfg = cv_score, (lam, coef)

# Plot coefficient magnitude vs. λ
plt.figure(figsize=(8,4))
plt.semilogx(lambdas, np.abs(coeff_paths))
plt.xlabel('λ (regularisation strength)')
plt.ylabel('Coefficient magnitude')
plt.title('L1 coefficient paths')
plt.show()
----

The loop prints a table of **CV R² scores** and stores the coefficient vectors for later visualisation. The plot reveals the classic **sparsity transition**: as λ exceeds ≈ 0.01 many coefficients collapse to zero, while the truly informative features retain non‑zero weight.

Below is a concise summary table that a student might copy into a lab notebook:

[cols="1,1,1", options="header"]
|===
| λ (regularisation strength) | CV R² score | Coefficient pattern

| 0.001 | 0.84 | All 20 non‑zero, modest shrinkage
| 0.01  | 0.87 | 5 strong, 3 weak, rest near zero
| 0.1   | 0.85 | 5 strong, 15 exact zeros (sparse)
| 1.0   | 0.78 | Over‑shrinkage, under‑fitting
|===

**What‑if scenario** – suppose the validation set is tiny (e.g., 5 % of the data). The CV curve may appear noisy and the algorithm could mistakenly select λ = 1.0. In that case we would switch to **nested cross‑validation** or reserve a separate hold‑out set to confirm the choice, illustrating the importance of robust evaluation.

===== Key Points

* Live notebook: generate data, define log‑spaced λ grid, run CV.  
* `Trainer(regularizer='l1', lambda_=lam, cv_folds=5)` encapsulates L1 CV.  
* Plotting coefficient magnitude vs. λ visualises sparsity emergence.  
* Table summarises λ → CV R² → coefficient pattern (sparse vs. dense).  
* Practical tip: start with a log‑spaced λ grid (10⁻³ … 10¹).  
* If validation data are scarce, consider nested CV to avoid over‑optimistic selection.  

==== Philosophical Reflection

Occam’s razor in machine learning is not a vague aesthetic; it is **encoded as a prior** over the hypothesis space. An L1 penalty corresponds to a **Laplace prior** that places most probability mass at zero, encouraging many weights to vanish. An L2 penalty corresponds to a **Gaussian prior** that prefers small but non‑zero weights. Elastic Net mixes the two priors, letting the practitioner express a belief that “most features are irrelevant, but a few may have modest effects”.

The **bias‑variance trade‑off** is the quantitative expression of this prior belief. By increasing λ we deliberately inject bias—our model is forced to conform to the prior—but we reap a reduction in variance, which translates into more reliable predictions on unseen data. Cross‑validation becomes the empirical mechanism that **tunes the strength of the prior**: the λ that maximises validation performance is the one that best reconciles our simplicity bias with the data’s complexity.

Beyond predictive performance, **simplicity carries ethical weight**. Sparse L1 models are easier to audit: each non‑zero coefficient can be inspected, explained, and linked to domain knowledge, supporting transparency and fairness assessments. However, an overly aggressive prior may discard subtle but legally or socially important relationships (e.g., interactions that affect protected groups). Thus, regularisation must be calibrated **with domain‑specific stakes in mind**, not merely to chase the highest CV score.

In Lab 2 you will experience this tension firsthand: after implementing L1 regularisation, you will examine the resulting coefficient vector on the **Adult Income** fairness dataset, discuss which features survive, and reflect on whether the induced sparsity aligns with ethical desiderata such as *explainability* and *non‑discrimination*.

===== Key Points

* Regulariser = explicit prior (L1 → Laplace, L2 → Gaussian).  
* Bias‑variance trade‑off formalises Occam’s razor in statistical terms.  
* Cross‑validation empirically balances bias (prior strength) and variance.  
* Sparsity improves interpretability and supports ethical auditing.  
* Over‑regularisation can hide genuine, fairness‑relevant patterns.  
* Lab 2 ties prior selection to real‑world interpretability challenges.
