*Adapt the agent for the specific problem.*

==== Hook: A Situation That Demands More Than a Generic Bot

Imagine a regional hospital that has just installed a conversational triage assistant. The base student‑AI agent can understand natural language, retrieve documents from a generic knowledge store, and call a handful of pre‑wired tools such as a web search and a simple calculator. On its first night shift, a nurse asks the bot, “Does Mr. Liu need an immediate CT scan for his suspected stroke?” The generic agent replies with a polite definition of a CT scan and a link to a Wikipedia article—useful for a layperson, but useless for a clinician who needs the patient’s latest lab values, the hospital’s imaging protocol, and a decision‑support rule that flags contraindications. The conversation stalls, the nurse escalates to a human, and the promise of AI‑augmented triage collapses.

The tension is clear: a *generic* language model is a powerful substrate, but without *domain‑specific prompts, tools, and data* it cannot act as a competent specialist. In this lecture we will answer the question, **“How do we transform a generic student‑AI agent into a situated, problem‑solving artifact?”** The answer unfolds through three intertwined layers—prompt engineering, tool integration, and knowledge injection—each of which we will design, implement, and evaluate.

==== Example Prompts (A Mini‑Demo)

When a user types a high‑level customization request, the system should immediately surface a concrete checklist that guides the next steps. For example:

```
User: Customize the agent for our emergency‑medicine triage workflow.
System: ✅ Prompt layer – define system prompt & few‑shot examples.
        ✅ Tool layer – register LabResultsAPI, ImagingProtocolAPI, RiskScoreCalculator.
        ✅ Data layer – index latest clinical guidelines and patient encounter cache.
        ✅ Integration – verify rate limits, latency budgets, and audit logging.
        ✅ Test – run unit, integration, and clinician‑acceptance scenarios.
```

This short vignette illustrates that *customization* is not a single button press but a series of deliberate actions that map directly onto the three layers we will explore.

==== Conceptual Core

The student‑AI agent delivered in the first half of the course is deliberately *generic*: it ships with a core LLM, a minimal orchestrator, and three built‑in tool interfaces (web search, calculator, and a simple document retriever). This genericity is a strength because it lets every cohort start from the same baseline, but it also means that the agent knows nothing about the particular regulatory constraints, data formats, or decision‑making heuristics of any real‑world domain.

**Customization** therefore becomes the design activity that situates the agent. We distinguish three orthogonal knobs:

1. **Prompt Layer** – The system prompt defines the agent’s role, tone, and high‑level policy; few‑shot examples illustrate the patterns of interaction that are expected in the target domain. For the hospital triage bot, a system prompt might read, “You are a certified emergency‑medicine assistant. You must always verify the patient’s latest vitals before recommending imaging.” Few‑shot examples would then show a dialogue where the bot asks for lab results, interprets them, and decides whether a CT is indicated.

2. **Tool Layer** – Domain‑specific APIs or wrappers extend the agent’s action space. In the triage scenario we might add a `LabResultsAPI` that returns the most recent blood‑gas values, an `ImagingProtocolAPI` that encodes hospital‑specific ordering rules, and a `RiskScoreCalculator` that implements the NIH Stroke Scale. Each tool is registered with the orchestrator via a small descriptor that specifies its name, input schema, and cost model.

3. **Data Layer** – Knowledge that the agent can retrieve on‑demand must be indexed in a memory store or knowledge graph. For a medical deployment this includes a curated set of clinical guidelines, a local drug‑interaction database, and a cache of recent patient encounters. The data layer is the only part that persists across sessions and can be refreshed without redeploying the model.

These three layers are not merely configuration steps; they are *design decisions* that shape the agent’s affordances and constraints. The process of customization therefore mirrors the classic engineering workflow: **requirements → analysis → specification → implementation → validation**. The agent lives inside an ecosystem of APIs, security firewalls, and cost‑governance policies, so integration concerns must be addressed early. Rate limits, latency budgets, and data‑privacy regulations dictate which tools can be called synchronously and which must be batched or cached.

> **Think‑Pair‑Share (2 min):** Choose a domain you are familiar with (e.g., finance, education, logistics). Sketch a quick “customization map” that lists one prompt tweak, one tool you would need, and one data source you would index. Share with your neighbor and note any overlaps.

Testing follows the same layered logic. **Unit tests** verify that each tool wrapper conforms to its schema and returns plausible mock data. **Integration tests** stitch the orchestrator, prompts, and tools together to simulate end‑to‑end conversations. **User‑acceptance tests** involve domain experts (e.g., clinicians) who run realistic scenarios and judge whether the agent’s recommendations are safe, accurate, and aligned with workflow. By iterating through these tests we move the agent from a *general* artifact to a *situated* one that respects the nuances of its operational context.

Finally, we must adopt a **design‑as‑customization mindset**: rather than treating the agent as a black box to be toggled on or off, we view each knob as a lever that can be refined, replaced, or removed as the problem domain evolves. This mindset prepares students for the inevitable need to revisit prompts, swap tools, or re‑index data when the deployment environment changes.

===== Key Points

* Prompt layer: system prompt + few‑shot examples  
* Tool layer: register domain‑specific APIs, wrappers, cost models  
* Data layer: index guidelines, patient records, knowledge graph  
* Integration constraints: rate limits, latency, security, cost  
* Testing hierarchy: unit → integration → user‑acceptance  
* General → situated artifact through design decisions  
* Customization is an iterative design process, not a one‑off configuration  

==== Technical Example (Legal‑Document Summarizer)

To show that the same customization workflow applies across domains, let’s walk through a corporate‑compliance use‑case: **a legal‑document summarizer for a compliance team**. The generic agent can read text and produce summaries, but the team needs summaries that flag regulatory clauses, reference internal policy IDs, and respect document confidentiality.

**1. Gap Analysis** – The team lists required capabilities: (a) a `ClauseClassifier` that tags sections as “Data‑Privacy”, “Financial‑Reporting”, or “Intellectual‑Property”; (b) a `PolicyLookup` service that maps clause tags to internal policy documents; (c) a `RedactionTool` that removes personally identifiable information before any output is shown. The generic agent lacks (a) and (c), and its knowledge base does not contain the firm’s policy IDs.

**2. Adding Custom Tools, Prompt, and Routing** –  
*Tool implementation* – A thin Python wrapper around the firm’s existing clause‑classification micro‑service:

```python
# file: tools/clause_classifier.py
from student_ai.orchestrator import ToolBase
import requests

class ClauseClassifier(ToolBase):
    name = "clause_classifier"
    description = "Classifies legal clauses into regulatory categories."

    def run(self, text: str) -> dict:
        resp = requests.post(
            "https://api.corp.com/classify",
            json={"document": text},
            timeout=5,
        )
        resp.raise_for_status()
        return resp.json()   # {"sections": [{"id":1,"tag":"Data-Privacy"}, …]}
```

*Registration* – The wrapper is declared in `routes.yaml`:

```yaml
tools:
  - module: tools.clause_classifier.ClauseClassifier
    enabled: true
    cost: 0.02   # USD per 1 000 tokens
```

*Prompt engineering* – The system prompt now frames the agent as a “Compliance‑aware summarizer” and explicitly instructs it to invoke the `clause_classifier` before drafting a summary:

```
You are an AI assistant that summarizes corporate legal documents.
First, call the tool `clause_classifier` on the full text.
Then, for each identified clause, retrieve the corresponding internal policy
using the `policy_lookup` tool. Finally, produce a concise summary that
highlights any compliance risks and references the policy IDs.
```

A few‑shot example demonstrates the desired flow:

```
User: Summarize the attached NDA.
Assistant: (calls clause_classifier) → receives tags...
Assistant: (calls policy_lookup) → retrieves policy 42...
Assistant: Summary: The NDA contains a data‑privacy clause (Policy 42) …
```

*Routing* – The orchestrator’s routing table encodes the workflow:

```yaml
routes:
  - trigger: "^summarize"
    steps:
      - tool: clause_classifier
      - tool: policy_lookup
      - tool: redaction_tool
      - output: final_summary
```

**3. End‑to‑End Test and Iteration** – A representative task is scripted:

```bash
python run_task.py --task "summarize" --file contracts/nda_v3.pdf
```

The test asserts that (a) every clause receives a tag, (b) the correct policy IDs appear in the output, and (c) no raw PII is present. Failures are logged, the prompt is tweaked, and the tool wrappers are refined until the suite passes. This hands‑on loop shows how a handful of targeted customizations—one new tool, a revised system prompt, and a routing rule—can transform a generic language model into a domain‑expert system that satisfies real business requirements.

===== Key Points

* Conduct a gap analysis to identify missing capabilities  
* Implement and register a custom tool (`ClauseClassifier`)  
* Engineer a system prompt that orchestrates tool calls  
* Define routing rules that encode the workflow sequence  
* Write an end‑to‑end test that validates the representative task  
* Iterate on prompts, tool wrappers, and routing until the test passes  

==== Philosophical Reflection

The transition from a *generic* to a *situated* agent is more than a technical checklist; it is an act of **designing an artifact into a social context**. Latour’s actor‑network theory reminds us that every tool we attach to the agent becomes part of a network of actors—human users, legacy databases, regulatory bodies, and even the cost‑governance policies that limit API calls. By choosing which tools to embed, we implicitly decide whose knowledge counts and whose constraints are respected.

Customization therefore carries **ethical responsibility**. When we inject proprietary policy documents or patient data into the agent’s knowledge layer, we inherit obligations for privacy, provenance, and bias mitigation. As Dr. Maya Patel, a hospital chief medical officer, warned after a pilot failure, “We trusted the bot’s recommendation because it sounded authoritative, but we never verified that the underlying lab values were up‑to‑date—privacy and safety slipped through the cracks.” Designers must therefore document the rationale behind each prompt, tool, and data source, and establish governance mechanisms (audit logs, versioning, consent records) that make the artifact accountable.

Finally, a situated agent is a **living artifact**. APIs evolve, regulations change, and user practices shift, causing inevitable drift. Recognizing this, we must adopt continuous stewardship: monitor performance metrics, re‑prompt or re‑train when drift is detected, and retire tools that become obsolete. In the next lecture we will explore concrete techniques for **monitoring, drift detection, and automated governance** of customized agents, completing the lifecycle from design to long‑term operation.

===== Key Points

* Customization embeds the agent in a sociotechnical network (Latour)  
* Ethical responsibility: privacy, provenance, bias mitigation, auditability  
* Real‑world quote illustrating stakes of poor customization  
* Drift awareness: continuous monitoring, re‑prompting, de‑commissioning  
* Ongoing stewardship bridges design and operations
