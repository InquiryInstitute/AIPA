=== 8.8: Lab Integration — Inference Engine
:leveloffset: 2

Imagine an AI tutor that must decide whether a student's answer to a geometry problem is *provably* correct before offering feedback.  
If the system relies only on a large‑language model to retrieve facts or to generate a natural‑language explanation, it may congratulate a wrong answer or flag a correct one as uncertain.  

*Failure case*: a student submits the statement “∠ABC = 90°” for a triangle that is actually obtuse. The LLM, having seen many examples of right‑angle proofs, confidently replies “Correct! Your reasoning is sound.” The tutor then awards full credit, reinforcing a misconception.  

Only a dedicated inference engine can *assert* premises, *query* relationships, and *prove* conclusions, turning raw retrieval into disciplined reasoning that catches such errors before they reach the learner.

==== Conceptual Core

The cognitive triad that powers an autonomous learner consists of **memory** (store and retrieve facts), a **large language model** (produce fluent text), and a **reasoning engine** (perform inference).  
The reasoning engine supplies the missing *structured‑thought* layer: it transforms retrieved symbols into logical deductions, probabilistic assessments, or chain‑of‑thought prompts that the LLM can elaborate.

Why does inference matter for safety?  By requiring a proof before an action, the agent can avoid hallucinations and unintended behaviors.  In safety‑critical domains—education, medicine, robotics—this guardrail is essential, because a single erroneous inference can cascade into harmful outcomes.  

From a software‑architecture perspective the engine lives in `student‑ai/reasoning/`.  Its public API is deliberately tiny, yet each call embodies a distinct reasoning step:

. **`assert(statement)`** – add a logical axiom (e.g., a universal rule about learning).  
. **`query(pattern)`** – retrieve matching facts from the knowledge base.  
. **`prove(goal)`** – attempt a formal proof using resolution or tableau methods (propositional or first‑order).  
. **`infer_probability(query)`** – compute a Bayesian estimate when the knowledge is incomplete.  
. **`chain_of_thought(prompt)`** – translate a proof or probability estimate into a natural‑language CoT trace that the LLM can expand.

These functions are called by the higher‑level agent during the ReAct loop (Chapter 9).  The engine therefore acts as a *submodule* that both consumes memory facts and produces prompts for the LLM, completing the triad.

**Design trade‑offs.**  A complete proof search guarantees soundness but can be exponential in the worst case; a probabilistic query is fast but yields only a confidence score.  In practice the engine first attempts a quick Bayesian estimate; only when the confidence exceeds a safety threshold does it launch the more expensive proof search.  This *dual‑mode* strategy lets a tutoring system respond in real time while still providing a rigorous audit trail when stakes are high.

**Real‑world case study.**  Consider a medical decision‑support assistant that must decide whether to recommend a biopsy.  A pure LLM might suggest “biopsy” based on surface patterns in the patient note, ignoring a subtle contraindication.  By asserting clinical guidelines as axioms and proving that “patient X satisfies contraindication Y,” the inference engine blocks the unsafe recommendation and forces the LLM to generate an explanation that references the guideline, dramatically increasing trustworthiness.

===== Key Points

* Cognitive triad: memory + LLM + reasoning engine  
* Failure case shows why raw LLM output is unsafe  
* Engine provides structured thought, safety guardrails, and dual‑mode reasoning (probability → proof)  
* API entry points: `assert`, `query`, `prove`, `infer_probability`, `chain_of_thought`  
* Located in `student‑ai/reasoning/` as a reusable submodule  
* Connects memory (facts) ↔ reasoning (deduction) ↔ LLM (natural language)  
* Enables the agent to prove goals before acting (see Chapter 9)  
* Supports deterministic logic, Bayesian inference, and hybrid orchestration  

==== Technical Example

Below is a minimal, runnable Python‑style snippet that demonstrates the full flow from assertion to LLM‑generated explanation, including a probabilistic query and an error‑handling path.

[source,python]
----
from reasoning import Engine, ProofError
from llm import LLM

engine = Engine()
llm = LLM()

# 1️⃣ Populate the knowledge base
engine.assert("forall x (Student(x) -> Learns(x))")
engine.assert("Alice is a Student")
engine.assert("forall x (Learns(x) -> ProbKnows(x, 0.9))")   # Bayesian prior

# 2️⃣ Quick probabilistic check
prob = engine.infer_probability("ProbKnows(Alice, p)")
print(f"Initial confidence Alice learns: {prob:.2f}")

# 3️⃣ Attempt a formal proof
goal = "Learns(Alice)"
try:
    proof = engine.prove(goal)          # may raise ProofError
    # 4️⃣ Turn the proof into a chain‑of‑thought prompt
    prompt = engine.chain_of_thought(f"Explain why {goal} holds.")
    explanation = llm.generate(prompt)
    print("Proof steps:", proof)
    print("LLM explanation:", explanation)
except ProofError as e:
    # fallback: let LLM reason from the probability
    fallback_prompt = f"Alice has a {prob:.0%} chance of learning. Explain why that may be sufficient."
    print("Proof failed:", e)
    print("LLM fallback explanation:", llm.generate(fallback_prompt))
----
**Setup** – The engine is instantiated and seeded with two axioms (a universal rule and a concrete fact) plus a Bayesian prior that every learner has a 90 % chance of truly understanding the material.

**Execution** – `infer_probability` gives an immediate confidence estimate.  If the confidence is already high, the system may skip the expensive `prove` call; otherwise it attempts a formal proof.  Successful proofs return a list of inference steps; failures raise `ProofError`.

**Verification** – Whether the proof succeeds or not, the result is handed to `chain_of_thought`, which crafts a natural‑language prompt for the LLM.  The LLM then produces a human‑readable explanation that can be shown to the student or clinician.

The snippet illustrates the three‑stage pipeline that the agent will repeat inside the ReAct loop: **retrieve** → **reason** → **generate**, with a probabilistic shortcut when appropriate.

===== Key Points

* Sample code shows full pipeline: assert → infer_probability → prove → CoT → LLM  
* `prove` returns a structured proof object (list of inference steps) or raises `ProofError`  
* `infer_probability` provides a fast confidence estimate that can short‑circuit proof search  
* `chain_of_thought` bridges formal output and natural language  
* Error handling demonstrates graceful fallback to probabilistic reasoning  
* Performance tip: cache intermediate results (e.g., proof of `Learns(Alice)`) to stay under real‑time constraints  
* Unit‑test example included in the lab (assert‑prove‑explain‑fallback test case)  

==== Philosophical Reflection

Reasoning in autonomous agents raises a classic tension between **rule‑following** and **rule‑breaking**.  Wittgenstein reminds us that “to reason is to follow a rule—or to break one consciously.”  An agent that always obeys formal logic may be brittle, while one that occasionally overrides a proof to act on intuition risks unreliability.  The inference engine therefore embodies a *deliberate* balance: it enforces rules when they are available, yet defers to the LLM’s probabilistic judgment when the knowledge base is incomplete.

From an ethical standpoint, reasoning acts as a **moral guardrail**.  By demanding a proof before a consequential action—such as assigning a grade or recommending medical treatment—the system makes its decision process transparent and auditable.  Probabilistic inference (`infer_probability`) acknowledges uncertainty, preventing the agent from presenting false certainty.  Together they form a *graded* accountability: high‑certainty situations trigger full proofs; low‑risk contexts rely on fast Bayesian estimates.

Looking ahead, hybrid formalisms that blend deductive logic, Bayesian reasoning, and generative language models promise richer cognition.  Yet limits remain: formal proof systems struggle with commonsense knowledge, while LLMs can hallucinate.  The future of AI reasoning lies in **orchestrating** these strengths, not in seeking a single universal formalism.  As we integrate the engine into the full ReAct loop (Chapter 9), we will see how structured thought becomes the infrastructure for responsible, adaptable action.

===== Key Points

* Rule‑following vs. rule‑breaking: when to trust formal proof vs. intuition  
* Reasoning as an ethical guardrail: transparency, auditability, and graded accountability  
* Probabilistic inference acknowledges uncertainty, avoiding false certainty  
* Limits of pure logic (commonsense gaps) and pure LLM (hallucination)  
* Hybrid orchestration (logic + probability + language) as the path forward  
* Wittgenstein’s insight applied to autonomous agents  
* Structured thought as infrastructure for responsible action
