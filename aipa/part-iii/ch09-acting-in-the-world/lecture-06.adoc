==== Hook: A Runaway Assistant

Imagine you ask an LLM‑driven assistant to *plan a weekend trip to the coast*.  
The assistant replies:

* “Do you prefer a beach or a mountain?”  
* “Do you prefer a beach or a mountain?”  
* “Do you prefer a beach or a mountain?”  

and so on, never producing a final itinerary.  
In a different scenario the same assistant, left unchecked, invokes a `delete_file` tool on a production server because the prompt never said *“do not delete”*.  

Both outcomes are disastrous: the user is stuck in an endless clarification loop, or the system causes irreversible damage.  
What prevented the assistant from looping forever or from executing a forbidden operation?  The answer lies in the **control‑flow stack** we explore in this lecture.

==== Example Prompts

* `"Stop after 8 steps."` — _max‑step budget_  
* `"Only use the search and translate tools."` — _allow‑list_  
* `"Never call external APIs that write files."` — _guard‑rail_  
* `"If you hit the step limit, give me a summary of what you have so far."` — _graceful degradation_

==== Conceptual Core

**1. Why loops happen** – An autonomous LLM agent follows a *reason‑act‑observe* cycle.  When the model’s confidence in its next action falls below a threshold, it often asks for clarification or retries a failing tool.  Without an external stop condition the cycle can repeat indefinitely, especially when the prompt is ambiguous or the tool returns noisy errors.

**2. Max‑step limits as a time‑out** – The simplest safeguard is a hard counter: each iteration increments `step_counter`; when it reaches `MAX_STEPS` the loop aborts and the agent returns a partial result together with an explanatory note.  This guarantees termination, caps compute cost, and gives the user a predictable latency bound.

**3. Allow‑list / block‑list as an action‑space filter** – Before a tool is invoked the agent checks whether the tool name appears in an *allow‑list* (the only tools it may call) or a *block‑list* (explicitly forbidden operations).  The list can be static (e.g., `{"search","summarize","translate"}`) or dynamic, derived from policy files that differ per deployment.  By pruning the action space we eliminate entire classes of unsafe behavior (e.g., file deletion, shell execution).

**4. Rate limits as a quantitative guard** – Even permitted tools may be abused if called excessively.  A per‑tool counter that resets every minute enforces a *rate limit* (e.g., at most 5 `search` calls per minute).  When the limit is exceeded the agent logs the violation and skips the call, thereby preventing denial‑of‑service attacks and runaway billing.

**5. Guard‑rails as semantic validation** – Guard‑rails operate on two fronts: *pre‑execution* checks that arguments satisfy a schema (no empty queries, length limits, no prohibited patterns) and *post‑execution* checks that the tool’s output does not contain disallowed content (PII, copyrighted text, or policy violations).  If a guard‑rail fails, the agent either warns the user or substitutes a safe placeholder.

**6. Graceful degradation and human‑in‑the‑loop overrides** – When any limit is hit, the agent should not simply crash.  Instead it returns a *partial* answer, cites the reason (e.g., “step budget exhausted”), and offers the user a chance to intervene.  An explicit *override flag* lets an operator abort execution immediately, providing a manual safety valve.

**7. Interplay of autonomy and control** – The three mechanisms form a layered envelope: the step budget guarantees that reasoning terminates; the allow‑list narrows what the agent can do; guard‑rails ensure each permitted action is semantically sound.  Within this envelope the agent retains creative freedom, while the boundaries protect users and infrastructure.

===== Key Points

* Infinite loops stem from confidence‑mismatch and ambiguous prompts.  
* `MAX_STEPS` acts as a hard time‑out, guaranteeing termination.  
* Allow‑list / block‑list prune the tool action space.  
* Rate limits cap API usage and cost.  
* Guard‑rails validate inputs *and* filter outputs.  
* Graceful degradation returns a partial result with an explanatory note.  
* Human‑override flags enable immediate aborts.  
* Logging every violation supports auditability and continuous improvement.

==== Technical Example

The following pseudo‑code stitches together all four control mechanisms.  It is deliberately language‑agnostic; in Lab 3 you will translate it into the framework of your choice and instrument it with real logging.

[source,python]
----
MAX_STEPS   = 8
ALLOWLIST   = {"search", "summarize", "translate"}
RATE_LIMIT  = {"search": 5, "summarize": 3}   # calls per minute
OVERRIDE    = False                           # set by a human operator

step_counter   = 0
rate_counters  = defaultdict(int)

def guardrail_passes(tool, args):
    # ---- pre‑execution validation ----
    if not args.get("query"):
        return False, "Missing query"
    if len(args["query"]) > 200:
        return False, "Query too long"
    # add more domain‑specific checks here
    return True, ""

def post_guardrail(output):
    # ---- post‑execution validation ----
    if "PII" in output:
        return False, "Output contains personal data"
    return True, ""

def execute_tool(tool, args):
    # placeholder for real tool invocation
    return f"Result of {tool} with {args['query']}"

def agent_loop(user_request):
    global step_counter, OVERRIDE
    while step_counter < MAX_STEPS:
        if OVERRIDE:
            log_event("human_override")
            return "Execution aborted by operator."

        step_counter += 1
        tool, args = plan_next_action(user_request)   # returns (tool, args)

        # ---- Allow‑list check ----
        if tool not in ALLOWLIST:
            log_violation(step_counter, tool, "not_allowed")
            continue   # skip this iteration

        # ---- Rate‑limit check ----
        if rate_counters[tool] >= RATE_LIMIT.get(tool, float('inf')):
            log_violation(step_counter, tool, "rate_limit")
            continue

        # ---- Guard‑rail (pre) ----
        ok, msg = guardrail_passes(tool, args)
        if not ok:
            warn_user(msg)
            continue

        # ---- Safe execution ----
        result = execute_tool(tool, args)
        if not result:
            warn_user("Tool failed")
            continue

        # ---- Guard‑rail (post) ----
        ok, msg = post_guardrail(result)
        if not ok:
            warn_user(msg)
            continue

        # ---- Success path ----
        if is_final_answer(result):
            return result

        # update internal state, possibly adjust next action
        update_state(result)

    # ---- Step budget exhausted ----
    return f"Partial result after {MAX_STEPS} steps: {summarize_progress()}"
----

==== Philosophical Reflection

**1. Agency under constraint** – From an *instrumental* viewpoint, limits are simply part of the agent’s utility function: without a budget the agent cannot achieve its goal efficiently because it wastes time or harms stakeholders.  From a *moral* perspective, however, the act of drawing a line is a normative judgment about what the system *ought* to be allowed to do.  Designers therefore become de‑facto policymakers, encoding societal values into `MAX_STEPS`, `ALLOWLIST`, and guard‑rail policies.

**2. Who decides the limits?** – In a medical‑diagnosis task a regulator might mandate a maximum of five reasoning steps before the system must defer to a human clinician.  In a financial‑advice scenario a product team might choose a higher step budget to allow deeper analysis.  The locus of responsibility shifts from the algorithm to the humans who configure its constraints, raising questions about accountability, transparency, and democratic oversight.

**3. Transparency as trust** – When the assistant says, “I stopped after 8 steps because the budget was exhausted; here is what I have so far,” the user perceives a *controlled failure* rather than an unexplained crash.  This openness lets stakeholders critique the chosen limits, propose adjustments, and ultimately align the system’s behavior with evolving ethical standards.

**4. Adaptive versus static limits** – Some argue for *dynamic* budgets that grow with task complexity, while others warn that adaptive limits can be gamed by adversarial prompts.  The tension between flexibility and predictability is at the heart of ongoing AI governance debates.

**5. The future of guard‑rails** – As models become more capable, guard‑rails may evolve from hand‑crafted rules to learned classifiers that detect policy violations.  Yet the fundamental question remains: *who* trains those classifiers, and *what* values do they encode?

===== Key Points

* Limits embed normative judgments into the agent’s design.  
* Responsibility for safety rests on the humans who set the constraints.  
* Transparent explanations of limit violations build user trust.  
* Static vs. adaptive budgets reflect a trade‑off between predictability and flexibility.  
* Future guard‑rails may be learned, but their training data remain a policy decision.

==== Synthesis

By weaving together a hard step budget, a curated allow‑list, per‑tool rate limits, and layered guard‑rail validation, we give autonomous agents a robust safety envelope.  The envelope guarantees termination, narrows the action space to safe operations, and enforces semantic correctness at every call.  Within these boundaries the agent retains creative problem‑solving power, while users gain predictability, trust, and a clear avenue for oversight.  In the next lecture we will see how a higher‑level governance layer can *dynamically* adjust these policies across deployments, turning static guard‑rails into a living policy engine.
