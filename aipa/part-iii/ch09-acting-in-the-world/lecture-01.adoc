=== 9.1: Agents — Perception, Reasoning, Action
:leveloffset: 2

[.epigraph]
"An agent is an entity that perceives and acts."
— Russell & Norvig, _AIMA_ <<RussellNorvig2020>>

*Hook.* You ask your virtual assistant, “Book me a flight to Tokyo next Friday.” Within seconds the system has parsed your spoken request, consulted your calendar, queried airline APIs, and returned a confirmed reservation. But what invisible machinery turned that single utterance into a concrete action in the world? The answer lies in a *problem‑driven* view of the **agent‑environment loop**: the assistant must *handle uncertainty*, *remember context*, and *plan a multi‑step interaction* before it can act. In the next 90 minutes we will trace a single dialogue turn by turn, watch how the agent’s internal state evolves, and ask at each step: *what does the agent need to know before it can safely act?*

==== Conceptual Core

The flight‑booking scenario forces us to ask a concrete question: **How can an agent act when it does not see the whole world and when the task unfolds over several turns?**  Imagine the following three‑turn exchange:

. *User*: “I need a flight to Tokyo.”  
. *Agent*: “When would you like to travel?”  
. *User*: “Next Friday.”

In turn 1 the agent receives only a partial request (“flight to Tokyo”) and must decide what to do next.  In turn 2 it asks a clarifying question, thereby *generating a new percept* that will be fed back into the loop.  In turn 3 the missing slot (date) is filled, allowing the agent to invoke the airline API.  This loop illustrates three essential ingredients:

* **Perception** – the raw utterance is turned into a structured *percept* (e.g., a set of extracted slots).  
* **Reasoning** – the agent evaluates which slots are still unknown, estimates the utility of possible actions (ask‑date vs. search‑flights), and selects the most promising one.  
* **Action** – the chosen function (`ask_date()` or `search_flights()`) is executed, which in turn changes the world and produces a new percept.

[.formal-definition]
****
*Key Idea (Agent)*  
An agent continuously receives **percepts** from its environment and maps the *history* of percepts to an **action**.  The mapping may be a simple rule (reflex) or a deliberative process that updates a **world model**, evaluates **utility**, and selects the action with the highest expected utility.  This view subsumes the formal definition in AIMA <<RussellNorvig2020>> while emphasizing the *temporal* and *uncertain* aspects that matter in practice.
****

The **environment** can be described along three orthogonal dimensions (see Table 1).  Each dimension determines which agent architecture is viable and why.

[cols="1,1,1", options="header"]
|===
| Dimension | Fully observable | Partially observable
| Deterministic | The thermostat (temperature known) | Simple robot in a known maze
| Stochastic | Slot‑machine (outcome random but state known) | Voice assistant (speech recognition noise)
| Episodic | Spam filter (each email independent) | Multi‑turn dialogue (current turn influences future)
|===

*Table 1 – Environment dimensions and illustrative examples.*

In a **partially observable, stochastic, non‑episodic** setting—exactly the case for our voice assistant—the agent cannot rely on a single percept to decide.  It must maintain a *belief state* (a probability distribution or a set of concrete slots) that aggregates evidence over time.  This requirement is the primary motivation for **model‑based** agents.

**Rationality** provides the normative guide: an agent should choose the action that maximizes its *expected utility* given its current belief about the world.  Utility encodes the agent’s goals (e.g., “book a flight that satisfies the user’s constraints”) and the expectation accounts for stochastic outcomes (e.g., flight‑price fluctuations, API latency).  When the belief is uncertain, the expected utility calculation naturally favors actions that *reduce uncertainty* (asking for the missing date) before actions that *consume resources* (searching for flights).

**Agent taxonomy.** Reflex agents implement a direct *percept → action* mapping; they are fast but cannot maintain hidden state or plan ahead.  Model‑based (deliberative) agents keep an internal **state / world model**, update it from each percept, and run a **planner** that evaluates possible action sequences against the utility function.  Table 2 summarizes the trade‑offs.

[cols="1,1,1,1", options="header"]
|===
| Feature | Reflex | Model‑based
| Memory | None | Belief state
| Planning horizon | 0 (instant) | Variable (look‑ahead)
| Handles uncertainty | Poor | Good (via belief update)
| Suitable tasks | Simple, fully observable | Sequential, partially observable
|===

*Table 2 – Reflex vs. model‑based agents.*

The **agent‑environment loop** is therefore a *closed feedback system*: every action produces a new percept, which may confirm a hypothesis or reveal a new unknown.  The loop continues until the utility of any further action falls below a threshold (e.g., the user says “Never mind”).  Understanding this loop is the prerequisite for designing the higher‑level *orchestrator* introduced in Chapter 10.

===== Key Points

* Agents must *perceive*, *reason*, and *act* in a continual feedback loop.  
* Partial observability forces the agent to maintain a belief state.  
* Rationality = maximizing expected utility under uncertainty.  
* Reflex agents: no internal state, instant response, limited to fully observable tasks.  
* Model‑based agents: maintain a world model, plan over a horizon, handle stochasticity.  
* The three environment dimensions (observability, determinism, episodicity) dictate the appropriate architecture.  
* The loop’s feedback nature makes *information‑gathering actions* (e.g., asking for a date) rational.  
* Tables 1 & 2 give a quick diagnostic for choosing an architecture.  
* The upcoming lab will let you turn these abstract components into concrete tool schemas.

==== Technical Example

**Reflex agent (percept → action).** The code below shows a bare‑bones rule‑based system that reacts only to the current utterance.

[source,python]
----
def reflex_agent(percept: str) -> str:
    """
    Very simple rule‑based agent.
    Percept: raw user utterance.
    Action: name of a function the orchestrator will call.
    """
    if "hello" in percept.lower():
        return "greet_user()"          # direct mapping
    if "flight" in percept.lower():
        # No memory of date or destination → fails gracefully
        return "ask_for_details()"
    return "do_nothing()"
----
*Explanation.* The agent examines the incoming string, applies a single rule, and returns the name of an actuator.  Because it never stores previous turns, the second rule cannot distinguish “I need a flight” from “I already gave the date”.  In a three‑turn dialogue the reflex agent would repeatedly emit `ask_for_details()` after each user response, never progressing to a search.

**Model‑based agent (state update + planner).** The following implementation keeps a belief dictionary, updates it from each percept, computes a tiny utility, and selects the next action.

[source,python]
----
class ModelBasedAgent:
    def __init__(self):
        self.belief = {}                     # internal world model

    def update_belief(self, percept: str):
        # Very naive NLU → extract key‑value pairs
        low = percept.lower()
        if "flight" in low:
            self.belief["task"] = "book_flight"
        if "tokyo" in low:
            self.belief["destination"] = "Tokyo"
        if "next friday" in low:
            self.belief["date"] = "2026-03-05"

    def evaluate_utility(self, action: str) -> float:
        # Simple utility: +1 for each filled slot
        filled = sum(1 for k in ("destination","date") if k in self.belief)
        return float(filled)

    def plan(self) -> str:
        # Choose action that maximizes utility
        if self.belief.get("task") != "book_flight":
            return "ask_for_task()"
        if "destination" not in self.belief:
            return "ask_destination()"
        if "date" not in self.belief:
            return "ask_date()"
        return "search_flights()"           # all slots filled → act

    def act(self, percept: str) -> str:
        self.update_belief(percept)
        chosen = self.plan()
        util = self.evaluate_utility(chosen)
        print(f"Belief: {self.belief} | Planned: {chosen} | Utility: {util}")
        return chosen
----
*Walk‑through.*  

1. **Turn 1** – User says “I need a flight to Tokyo.”  
   *`update_belief`* stores `task=book_flight` and `destination=Tokyo`.  
   *`plan`* sees that `date` is missing and returns `ask_date()`.  
2. **Turn 2** – User replies “Next Friday.”  
   *`update_belief`* adds `date=2026‑03‑05`.  
   Now all slots are present, so `plan` returns `search_flights()`.  
3. **Turn 3** – The agent calls the airline API, receives a list, and presents it to the user.

The model‑based agent therefore *reduces uncertainty* before committing to a costly external call, embodying the rationality principle.

===== Key Points

* Reflex agents need no internal state; they are fast but brittle.  
* Model‑based agents maintain a belief state and a planner that selects actions by utility.  
* The belief‑update step corresponds to the *Percept Processor → State* arrow in the diagram.  
* The planner embodies *rationality* by maximizing expected utility (here approximated by filled slots).  
* Failure mode demo: a reflex agent loops on `ask_for_details()`; the model‑based agent progresses once the missing slot is supplied.  
* The utility function can be enriched (price, carbon footprint) without changing the overall architecture.  
* Mini‑design activity: **Think‑Pair‑Share** – sketch a new percept (e.g., “I need a hotel”) and predict how the model‑based agent would update its belief.

==== Philosophical Reflection

**Distributed agency.** When the model‑based agent calls `search_flights()`, it delegates the actual lookup to an external airline API.  The *agency* is therefore distributed across the reasoning core and the toolbox it commands.  In this view, the “agent” is not a monolithic entity but a *composite system* whose identity emerges from the tight coupling of perception, model, planner, and actuators.  This perspective raises a pressing question: *who is responsible when the API returns a misleading price or a flight that violates a user’s travel policy?*  Responsibility must be allocated across the AI developer (who designed the planner), the API provider (who supplies the data), and the end‑user (who authorizes the purchase).  Ignoring this distribution can lead to blame‑shifting and opaque failure modes.

**Goal design as ethical engineering.** The utility function we embedded—“book the cheapest flight that satisfies the user’s constraints”—encodes a value judgment.  Suppose a corporate user has a policy that caps carbon emissions at 1 t CO₂ per trip.  If the cheapest flight emits 2 t CO₂, a naïve utility maximizer would still select it, violating the policy.  To respect the policy we must *re‑weight* the utility:

```
U = -price   - λ * carbon_emission
```

where λ reflects the organization’s carbon priority.  Changing λ changes the agent’s behavior dramatically: a high λ forces the planner to discard cheap but dirty flights, while a low λ may ignore the policy altogether.  This simple algebraic tweak illustrates how *goal engineering* is a direct lever for ethical outcomes.  Moreover, the utility function should be *transparent* (documented, auditable) and *revisable* as regulations or stakeholder values evolve.

**Accountability loops.** Because the agent’s actions are mediated by external tools, we can insert *monitoring hooks* after each tool call.  A post‑hoc audit can compare the chosen action’s utility against a compliance checklist (price ceiling, carbon budget, user consent).  If the audit flags a violation, the orchestrator can trigger a *remediation* sub‑loop: inform the user, propose alternatives, or roll back the transaction.  This design makes the ethical safeguard an integral part of the feedback loop rather than an after‑thought.

===== Key Points

* Agency is *distributed* across the reasoning core and its toolset; the “agent” is a composite system.  
* Responsibility for failures must be shared among developers, tool providers, and users.  
* Utility functions encode normative choices; altering weights changes behavior (price vs. carbon).  
* Transparent, auditable goal specifications are a prerequisite for trustworthy agents.  
* Monitoring hooks after each tool invocation create an *accountability loop* within the agent‑environment feedback system.  
* Model‑based agents expose a clear insertion point for ethical constraints (during utility evaluation).  
* Reflex agents are sufficient only for fully observable, single‑step tasks where ethical nuance is minimal.
