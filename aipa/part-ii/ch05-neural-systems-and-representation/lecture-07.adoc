=== 7. Model Export and Representation

[.epigraph]
____
*When a self‑driving car’s perception stack crashed because the model could not be loaded on the edge device, the engineers discovered that the model had never been compiled for the target hardware.*  
How do we guarantee that a PyTorch model runs reliably on any device, from a cloud GPU to a tiny micro‑controller?
____

=== Hook

Imagine you have spent weeks training a state‑of‑the‑art image classifier in PyTorch. The training script runs flawlessly on your laptop, the validation accuracy is impressive, and you are ready to ship the model to a fleet of autonomous drones. You zip the `model.pth` file, copy it to the drone, and run the inference script—only to watch the drone stall, the CPU spike to 100 %, and the predictions become random. The culprit? The model was saved as a pure Python object, which depends on the exact version of PyTorch, the same Python interpreter, and a host of runtime libraries that the drone does not have.

This scenario raises a tension that sits at the heart of modern AI engineering: **portability versus transparency**. On the one hand, we want models that can be moved across platforms, compiled for speed, and executed in environments where Python is unavailable. On the other hand, we risk turning a transparent, inspectable Python model into an opaque binary that is harder to audit, debug, or modify. The tools that mediate this tension—TorchScript and ONNX—are the focus of today’s lecture.

=== Conceptual Core

TorchScript and ONNX are *intermediate representations* (IRs) that sit between the high‑level Python API and the low‑level execution engine. They freeze the computational graph, resolve dynamic Python control flow, and produce a portable artifact that can be loaded by a runtime written in C++, Java, or even WebAssembly.

* **TorchScript** is a PyTorch‑specific IR. It can be generated by *tracing* (recording tensor operations on example inputs) or by *scripting* (compiling Python code that contains control flow). The result is a serialized `.pt` file that contains both the graph and the compiled operators. Because the graph is static, the TorchScript runtime can apply optimisations such as operator fusion, constant folding, and memory planning, often yielding 2‑5× speed‑ups on CPUs and GPUs.

* **ONNX** (Open Neural Network Exchange) is an open, framework‑agnostic format. Exporters from PyTorch, TensorFlow, and many other libraries translate their models into a protobuf that describes nodes, tensors, and attributes. ONNX is the lingua franca for deploying models on specialised runtimes (e.g., TensorRT, OpenVINO) and on edge devices that lack a full PyTorch stack.

* **`dest`** is the destination folder or storage service where the exported artifact is written. In production pipelines, `dest` is usually a version‑controlled artifact repository (e.g., an S3 bucket with immutable keys, a Git LFS store, or a model‑registry service). Storing the model in a reproducible location decouples training from serving and enables continuous‑delivery workflows.

* **Portability** is achieved by eliminating the Python interpreter from the inference path. The compiled artifact can be loaded by a lightweight runtime that only needs the operator kernels and the serialized graph.

* **Safety** is enhanced because the graph is immutable; accidental changes to model weights or architecture after export are prevented by design.

* **Reproducibility** hinges on recording the exact versions of the compiler, the runtime, and the `dest` location. A mismatch can lead to subtle numerical differences or outright failures.

* **Trade‑offs**: TorchScript retains PyTorch‑specific semantics (e.g., custom autograd functions) but is less portable than ONNX. ONNX offers broader ecosystem support but may lose some PyTorch‑only features during export.

* **Lifecycle**: The typical lifecycle is *train → validate → export (TorchScript/ONNX) → store (`dest`) → deploy → monitor*.

.Key Points
* TorchScript freezes Python control flow into a static graph.
* ONNX provides a framework‑agnostic exchange format.
* `dest` must be a write‑able, version‑controlled location for reproducibility.
* Compilation enables runtime optimisations (fusion, constant folding).
* Portability removes the Python dependency from inference.
* Safety is increased by making the model immutable after export.
* Trade‑offs exist between PyTorch‑specific features and ecosystem breadth.
* A robust pipeline records compiler and runtime versions alongside the artifact.

=== Technical Example

Below is a minimal end‑to‑end example that demonstrates how a simple feed‑forward classifier can be exported both as TorchScript and as ONNX, saved to a destination folder, and later loaded in a pure C++ runtime.

[source,python]
----
import torch
import torch.nn as nn
import torch.nn.functional as F
import pathlib

class SimpleClassifier(nn.Module):
    def __init__(self, in_features=10, hidden=16, out_features=3):
        super().__init__()
        self.fc1 = nn.Linear(in_features, hidden)
        self.fc2 = nn.Linear(hidden, out_features)

    def forward(self, x):
        x = F.relu(self.fc1(x))
        # Conditional logic demonstrates scripting capability
        if x.mean() > 0.5:
            x = self.fc2(x)
        else:
            x = self.fc2(x) * 0.5
        return F.log_softmax(x, dim=-1)

# Instantiate and dummy‑train
model = SimpleClassifier()
optimizer = torch.optim.Adam(model.parameters(), lr=0.01)
dummy_input = torch.randn(4, 10)
target = torch.tensor([0, 1, 2, 1])

for _ in range(10):
    optimizer.zero_grad()
    loss = F.nll_loss(model(dummy_input), target)
    loss.backward()
    optimizer.step()
----

#### TorchScript Export

[source,python]
----
# Scripting preserves the conditional branch
scripted = torch.jit.script(model)

# Save to a temporary location
script_path = pathlib.Path("exports/simple_classifier_ts.pt")
scripted.save(str(script_path))

print(f"TorchScript model saved to {script_path}")
----

#### ONNX Export

[source,python]
----
onnx_path = pathlib.Path("exports/simple_classifier.onnx")
torch.onnx.export(
    model,
    dummy_input,
    str(onnx_path),
    export_params=True,
    opset_version=14,
    do_constant_folding=True,
    input_names=['input'],
    output_names=['log_probs'],
    dynamic_axes={'input': {0: 'batch_size'}, 'log_probs': {0: 'batch_size'}}
)
print(f"ONNX model saved to {onnx_path}")
----

#### Storing to `dest`

In a production pipeline, `dest` would be an artifact repository. For illustration we simply move the files into a version‑controlled directory.

[source,python]
----
import shutil
dest = pathlib.Path("artifact_repo/v1.0.0/")
dest.mkdir(parents=True, exist_ok=True)

shutil.move(str(script_path), dest / script_path.name)
shutil.move(str(onnx_path), dest / onnx_path.name)

print(f"Artifacts stored in {dest}")
----

#### C++ Inference (Skeleton)

```cpp
// main.cpp
#include <torch/script.h>
#include <iostream>

int main() {
    torch::jit::script::Module module;
    try {
        // Load the TorchScript artifact
        module = torch::jit::load("artifact_repo/v1.0.0/simple_classifier_ts.pt");
    } catch (const c10::Error& e) {
        std::cerr << "Error loading the model\\n";
        return -1;
    }

    // Create a dummy input tensor
    torch::Tensor input = torch::randn({4, 10});
    std::vector<torch::jit::IValue> inputs;
    inputs.push_back(input);

    // Run inference
    at::Tensor output = module.forward(inputs).toTensor();
    std::cout << "Log probabilities:\\n" << output << std::endl;
    return 0;
}
```

Compile with:

```bash
c++ -I/path/to/libtorch/include -I/path/to/libtorch/include/torch/csrc/api/include \
    main.cpp -L/path/to/libtorch/lib -ltorch -lc10 -o infer
```

Running the binary typically yields a **2‑3× latency reduction** compared to loading the original `model.pth` via Python, because the TorchScript runtime eliminates Python overhead and performs operator fusion.

.Key Points
* `torch.jit.script` preserves Python control flow; `torch.jit.trace` records only tensor operations.
* ONNX export requires explicit input names and dynamic axes for batch‑size flexibility.
* The `dest` folder should be immutable for a given version (e.g., `v1.0.0`).
* C++ inference loads the serialized `.pt` without importing Python.
* Runtime optimisations (fusion, memory planning) are applied automatically.
* Version mismatches between the exporter and the runtime cause load failures.
* Exported artifacts can be audited by inspecting the serialized graph (e.g., `torch.jit.load(...).graph`).
* Both formats support downstream quantisation and pruning pipelines.

=== Philosophical Reflection

The act of compiling a model into an opaque binary raises profound questions about **accountability** and **auditability** in AI systems. When a model is stored as a Python `state_dict`, any practitioner can open the file, inspect the weights, and even modify the architecture. This transparency aligns with the principle of *explainability*: stakeholders can trace predictions back to learned parameters.

Conversely, a TorchScript or ONNX artifact is deliberately *closed*: the graph is serialized in a format that is not meant for casual human reading. While this closure brings performance and portability, it also creates a barrier for external auditors who wish to verify that the model does not contain hidden biases or malicious logic. The tension mirrors broader debates in software engineering between *open source* and *proprietary* binaries.

Moreover, the choice of `dest` embeds a governance decision. Storing artifacts in a version‑controlled repository ensures **provenance**—every model version is linked to a specific training run, data snapshot, and compiler version. Yet, if the repository is managed by a single vendor, the community may lose the ability to reproduce the model independently, re‑introducing a form of *vendor lock‑in*.

These considerations suggest a set of ethical guidelines:

* **Transparency by Design**: Even when deploying compiled models, maintain a parallel human‑readable specification (e.g., a JSON description of the graph) that can be inspected by auditors.
* **Immutable Provenance**: Record the exact compiler, runtime, and `dest` metadata alongside the artifact; treat this metadata as part of the model’s legal contract.
* **Open Export Paths**: Prefer ONNX when cross‑framework interoperability is required, as it encourages a shared ecosystem rather than a single‑vendor runtime.
* **Audit Hooks**: Design runtime wrappers that log intermediate tensor statistics, enabling post‑hoc analysis without exposing the source code.
* **Responsible Decommissioning**: When a model is retired, ensure that its compiled artifact is archived with the same provenance guarantees as the original training artifacts.

Reflecting on these points helps students see that model export is not merely a technical step but a **sociotechnical decision** that influences trust, regulation, and the long‑term sustainability of AI systems.

.Key Points
* Compiled artifacts improve performance but reduce human readability.
* Provenance metadata (compiler version, `dest` path) is essential for reproducibility.
* ONNX promotes ecosystem openness; TorchScript is PyTorch‑centric.
* Auditable specifications should accompany binary models.
* Ethical deployment requires balancing portability with transparency.
* Governance of `dest` influences vendor lock‑in and community trust.
* Runtime logging can provide post‑hoc explainability without source exposure.
* Decommissioned models must be archived with full metadata.
