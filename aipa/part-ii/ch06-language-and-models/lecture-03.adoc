=== Conceptual Core

**Paragraph 1 – Autoregressive Generation as a Real‑Time Conversation**  
A causal language model (CLM) behaves like a conversational partner that can only speak after it has heard everything that came before. Concretely, given a token sequence *t₁ … tₙ*, the model computes a probability distribution *P(tₙ₊₁ | t₁…tₙ)* and samples the next token. This left‑to‑right constraint is what makes CLMs suitable for tasks where the output length is not known in advance, such as summarization.  

*Key points*  

* **Autoregressive prediction** – each token depends only on earlier tokens, never on future ones.  
* **Maximum‑likelihood training** – the model is optimized to assign high probability to the true next token across billions of text windows.  
* **Token‑by‑token inference** – generation stops when a special *<eos>* token appears or a user‑defined length limit is reached.

**Paragraph 2 – From Llama‑2 to Llama‑3: Sparsity Meets Scale**  
Llama‑3 keeps the familiar transformer stack but replaces the dense feed‑forward network (FFN) with a *Mixture‑of‑Experts* (MoE) module. Instead of every token flowing through a single 4 000‑dimensional matrix, a lightweight gating network selects 2 or 3 experts out of a pool of 8 to process that token. The selected experts compute their own transformations, and their outputs are summed. Because only a fraction of the parameters are active per token, the model can allocate more total capacity (8 B parameters) while staying within the memory budget of a single consumer GPU.  

*Key points*  

* **Sparse activation** – at inference time only a subset of experts (≈25 %) are evaluated, reducing FLOPs.  
* **Rotary positional embeddings (RoPE) 2.0** – extended to 4 096 dimensions, improving long‑range coherence for documents longer than 1 024 tokens.  
* **Expanded pre‑training corpus** – includes web snapshots up to 2023, cutting down the “temporal drift” that plagued Llama‑2 on recent events.  
* **Parameter efficiency** – an 8 B MoE model matches or exceeds the zero‑shot performance of a 13 B dense baseline on GLUE and summarization benchmarks.  

**Paragraph 3 – Tokenization: The Bridge Between Text and Numbers**  
Llama‑3 uses a byte‑pair encoding (BPE) tokenizer with a vocabulary of roughly 32 k subword units. The tokenizer first splits a word into the longest matching sub‑tokens; rare or out‑of‑vocabulary words are broken down into byte‑level pieces, guaranteeing that *any* Unicode string can be represented. Because the model works on IDs, the length of the tokenized prompt directly determines both GPU memory consumption and latency.  

*Key points*  

* **`AutoTokenizer.from_pretrained`** loads the vocab and merge rules in a single call.  
* **`skip_special_tokens=True`** removes padding, BOS, and EOS markers from the final decoded string, yielding clean human‑readable output.  
* **Token‑length budgeting** – a 2‑sentence summary of a typical news article fits in ≈ 80 tokens; exceeding this limit forces the model to truncate or generate an EOS early.  

**Paragraph 4 – Sampling Controls: Temperature, Top‑p, and Token Limits**  
During generation the model emits a vector of logits for the entire vocabulary. The *temperature* scalar (τ) rescales these logits: lower τ sharpens the distribution (more deterministic), higher τ flattens it (more random). *Top‑p* (nucleus) sampling retains the smallest set of tokens whose cumulative probability exceeds *p* (e.g., 0.9) and renormalizes them, discarding the long tail. Finally, `max_new_tokens` caps the number of tokens the model may emit before stopping.  

*Key points*  

* **`temperature=0.7`** – a sweet spot that reduces repetitive phrasing while keeping the output sensible.  
* **`top_p=0.9`** – guarantees that the model never chooses a token outside the 90 % most probable mass, limiting bizarre word choices.  
* **`max_new_tokens=80`** – empirically enough for a concise two‑sentence summary of most English articles.  
* **Interaction effect** – raising temperature while keeping top‑p high can produce creative but potentially inaccurate paraphrases; lowering both yields safe, “copy‑the‑source” style output.  

*Quick‑check*: Predict what happens if we set `temperature=0.2` and `top_p=0.8`. (Answer: the model will almost always pick the single most likely token at each step, producing a very deterministic, possibly repetitive summary.)

**Paragraph 5 – Resource Footprint, Quantization, and the “Enough?” Question**  
The full‑precision (16‑bit) checkpoint of Meta‑Llama‑3‑8B occupies ~30 GB of VRAM, far beyond a single RTX 4090. By applying 4‑bit *GPTQ* quantization the model shrinks to ~16 GB, enabling real‑time inference with latency under 2 seconds for a typical prompt. Quantization introduces a modest drop in perplexity (≈ 0.3 p pts) but preserves most downstream performance. The trade‑off—speed versus a slight loss of fidelity—is the core of the “Enough?” question: is an 8 B, quantized model sufficient for a newsroom that cannot afford hallucinations?  

*Key points*  

* **4‑bit quantization** – reduces memory and bandwidth, allowing a single‑GPU deployment.  
* **Latency scaling** – roughly linear with `max_new_tokens`; a 40‑token request finishes in ~1 s, 80 tokens in ~1.8 s.  
* **Accuracy impact** – small increase in token‑level error, but often imperceptible in short summaries.  
* **Operational simplicity** – no need for model‑parallelism or off‑loading, which lowers engineering overhead.

=== Technical Example: Walking Through the Code

Below is a minimal, reproducible script that loads Meta‑Llama‑3‑8B, tokenizes a prompt, generates a two‑sentence summary, and prints the result. The surrounding prose walks you through each logical block, showing not only *what* the code does but *why* each line matters for the constraints introduced in the hook.

[source,python]
----
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

# -------------------------------------------------
# 1️⃣ Load the model and tokenizer (4‑bit quantized)
# -------------------------------------------------
model_name = "meta-llama/Meta-Llama-3-8B"
tokenizer = AutoTokenizer.from_pretrained(model_name, use_fast=True)

model = AutoModelForCausalLM.from_pretrained(
    model_name,
    load_in_4bit=True,          # fits in ~16 GB VRAM
    device_map="auto",          # puts layers on GPU if available
    torch_dtype=torch.float16   # keeps arithmetic fast
)

# -------------------------------------------------
# 2️⃣ Craft a task‑oriented prompt
# -------------------------------------------------
article = """The city council voted yesterday to allocate $12 million for a new
public‑transport hub.  Proponents say the project will cut commute times by
15 % and create 2 000 jobs, while opponents warn of cost overruns and
environmental impact."""
prompt = f"""Summarize the following article in two sentences:\n\n{article}"""

# -------------------------------------------------
# 3️⃣ Tokenize – produces input_ids and attention_mask
# -------------------------------------------------
inputs = tokenizer(prompt, return_tensors="pt").to(model.device)

# -------------------------------------------------
# 4️⃣ Generate – sampling knobs in action
# -------------------------------------------------
output_ids = model.generate(
    **inputs,
    max_new_tokens=80,
    temperature=0.7,
    top_p=0.9,
    do_sample=True,
    pad_token_id=tokenizer.eos_token_id
)

# -------------------------------------------------
# 5️⃣ Decode – strip special tokens for a clean string
# -------------------------------------------------
summary = tokenizer.decode(output_ids[0], skip_special_tokens=True)
print("🗞️ Summary:", summary)
----
*Key points in the walkthrough*  

* **Quantized loading** (`load_in_4bit=True`) shrinks the checkpoint to a size that a single RTX 4090 can hold, directly addressing the resource constraints raised in the hook.  
* **`device_map="auto"`** automatically dispatches layers to GPU memory, falling back to CPU for any overflow, which keeps the script portable across hardware configurations.  
* **Prompt design** – the explicit “Summarize … in two sentences” instruction leverages Llama‑3’s instruction‑following fine‑tuning, improving adherence to length constraints.  
* **`do_sample=True`** activates stochastic sampling; without it the model would default to greedy decoding, often producing repetitive or overly terse output.  
* **`pad_token_id=tokenizer.eos_token_id`** ensures the model knows where to stop if it reaches the maximum token budget before emitting an EOS token.  

*Experimentation suggestions*  

1. **Determinism test** – set `temperature=0.2` and `top_p=0.8`; observe how the summary becomes almost verbatim to the source.  
2. **Creativity test** – raise `temperature` to 1.2 and `top_p` to 0.95; note occasional paraphrases or inserted adjectives.  
3. **Length stress test** – increase `max_new_tokens` to 150 and watch the model drift into a third sentence, revealing the limits of the “two‑sentence” instruction.

=== Philosophical Reflection: Risks, Bias, and the “Enough?” Question

Large language models are powerful amplifiers, but they are also prone to *hallucination*—the generation of statements that have no grounding in the source text. In a newsroom, a two‑sentence summary that invents a statistic (e.g., “15 % reduction in traffic”) when the original article only mentioned “significant reduction” can mislead readers and damage editorial credibility.  

*Key ethical considerations*  

* **Hallucination** – The model may fill gaps with plausible‑sounding but false details, especially when temperature is high.  
* **Bias propagation** – Training data reflects societal biases; Llama‑3 can reproduce gendered language (“the mayor, a man…”) even if the source is gender‑neutral.  
* **Safety mitigations** – Open‑source releases lack built‑in content filters; developers must implement profanity or disallowed‑content detectors.  
* **Transparency** – Ethical journalism demands a clear label (“Generated by Meta‑Llama‑3‑8B”) and a link to the full article for verification.  
* **“Enough?”** – The newsroom must decide whether an 8 B, quantized model offers a sufficient trade‑off between speed and reliability, or whether a larger, more accurate model (or a human‑in‑the‑loop) is required for high‑stakes reporting.  

*Case study*: During a pilot at a regional paper, Llama‑3 generated a summary that claimed “the council approved a $12 million budget” while the original brief only said “the council is *considering* a $12 million allocation.” The error stemmed from the model’s tendency to *complete* ambiguous statements when the temperature was set to 0.9. The editorial team responded by adding a post‑generation verification step—automatically comparing the generated summary against named entities extracted from the source.  

*Forward look*: In the lab you will experiment with temperature and top‑p to see how often such factual drift occurs, and you will implement a simple verification script that flags mismatched numbers. This hands‑on experience will help you answer the “Enough?” question with data, not intuition.
