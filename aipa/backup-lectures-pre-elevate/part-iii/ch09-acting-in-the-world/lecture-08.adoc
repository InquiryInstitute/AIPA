==== Hook: A Real‑World Scenario

Imagine a researcher asks the system: *“Find the latest papers on quantum‑error‑correction, summarise the key ideas, and generate a reproducible experiment plan.”*  
The request touches many capabilities: a web search, a memory‑augmented retrieval, logical reasoning about the papers, natural‑language generation, and finally an audit of the generated plan.  
How can a single agent coordinate all these tools without the student writing glue code for each step? The answer lies in the **tool‑calling agent**, the first fully‑closed cognitive loop in our curriculum.

==== Conceptual Core

*Recap of building blocks* – In Chapters 7‑9 we introduced six core tools: **search**, **memory**, **reasoning**, **llm**, **audit**, **ml_trainer**, and **neural_classifier**. Each tool follows a common **schema** (name, description, input‑type, output‑type) and is discovered through a **registration** file placed in `student‑ai/tools/`.

*ReAct cycle* – The agent follows the classic **think → act → observe** loop. After receiving a user prompt it *thinks* (generates a plan), *acts* by invoking the most appropriate tool, and then *observes* the tool’s result to update its internal state.

*Planning & replanning* – Complex tasks are decomposed into sub‑goals. If a tool call fails or yields unexpected output, the agent **replans** by revisiting earlier reasoning steps, possibly selecting an alternative tool.

*Control mechanisms* – To guarantee safety we enforce a **max‑step limit** and a **safety guard** that aborts execution on suspicious outputs (e.g., overly long responses, prohibited content). The **audit** tool records every tool call, its inputs, and its outputs for later inspection.

*Folder layout & orchestrator preview* – All tool implementations live under `student‑ai/tools/`. Registration files are automatically loaded at runtime. Chapter 10 will introduce the **orchestrator**, a higher‑level controller that can coordinate multiple such agents or delegate tasks among them.

===== Key Points

* tools schema, registration, and discovery  
* ReAct loop: think → act → observe  
* task decomposition and replanning strategies  
* safety guard: max‑step limit and abort criteria  
* audit logging of every tool call  
* folder structure: `student‑ai/tools/`  
* bridge to Ch10: orchestrator will manage multiple agents  

==== Technical Example

*End‑to‑end demo* – The student runs a script that sends the prompt *“Research topic X, summarize with citations, run an audit on your process.”* The agent first **searches** for relevant papers, stores the results in **memory**, uses the **reasoner** to extract key ideas, calls the **llm** to draft a summary, and finally invokes the **audit** tool to verify citation correctness and log the workflow.

*Audit sub‑step* – The audit tool writes a JSON entry for each call: `{ "tool": "search", "input": "...", "output": "...", "timestamp": "..." }`. This log is later used to generate a provenance report, satisfying transparency requirements.

*Failure recovery* – If the search service times out, the agent falls back to a cached knowledge base stored in memory. It then proceeds with the remaining steps, annotating the final answer with a “fallback used” flag. This demonstrates robust replanning.

*Performance metrics* – Students instrument the demo to capture **latency**, **token cost**, and **error rate** for each tool call, enabling quantitative evaluation of the integrated system.

===== Key Points

* end‑to‑end demo: search → memory → reason → llm → audit  
* audit log format and provenance report  
* fallback to cached knowledge on search failure  
* performance metrics: latency, token cost, error rate  
* testing harness: unit tests for each tool call  
* safety controls: max‑step limit, abort on suspicious output  
* submission script that reproduces the demo  

==== Philosophical Reflection

*Agency vs. tool autonomy* – The agent “acts” only by delegating to external tools. This raises the question: does agency reside in the decision‑making component, in the tools themselves, or in the emergent behaviour of the whole system? Understanding this distinction informs how we design accountability mechanisms.

*Ethical dimensions* – Tool‑calling introduces provenance and bias concerns. The **audit** tool becomes a guardrail, ensuring that generated content can be traced back to its sources and that any downstream decisions respect ethical standards. Students must consider who bears responsibility when a tool produces harmful output.

*Transparency and human‑in‑the‑loop* – By exposing the audit log, the system invites human oversight. This aligns with the broader goal of **human‑in‑the‑loop AI**, where users can intervene, correct, or halt the agent’s actions based on observed behaviour.

===== Key Points

* agency resides in decision‑making, not merely tool execution  
* responsibility allocation between agent and tools  
* audit logs provide transparency and provenance  
* bias mitigation through tool selection and audit checks  
* human‑in‑the‑loop design encourages oversight and correction
