=== 3.5: Planning as Search — STRIPS, PDDL
:leveloffset: 2

Imagine a robot in a cluttered workshop that must move three wooden blocks from a scattered table into a neat tower.  
Which sequence of grasps, lifts, and placements will achieve the goal without dropping any block?  
Behind this simple puzzle lies a massive combinatorial search problem that planners solve every day.

*Planning is search in the space of possible action sequences—with structure.*

==== Conceptual Core

pass:q[\aimaterm{planning}] formulates problems as **states** (conjunctions of facts), **actions** (preconditions and effects), an **initial state**, and a **goal**.  
The **STRIPS** (Stanford Research Institute Problem Solver) representation captures this structure: an action is applicable in a state if its preconditions hold; applying it produces a new state by adding and deleting effects.

**Forward (progression) planning** searches from the initial state, repeatedly applying applicable actions until a goal state is reached.  
**Backward (regression) planning** starts from the goal and works in reverse, selecting actions whose effects satisfy the current sub‑goals, ultimately reaching the initial state.

**Plan‑space search** operates at a higher level: instead of enumerating concrete states, it refines *partial plans* (sets of actions with ordering constraints) until a complete, conflict‑free plan emerges.

The **frame problem** asks what stays unchanged when an action occurs. Classical planning adopts the **frame axiom**: only the facts explicitly listed in an action’s add‑ and delete‑lists change; everything else persists. This assumption simplifies search but breaks down in domains with indirect or continuous effects.

**PDDL** (Planning Domain Definition Language) standardizes how STRIPS‑style domains are described. A *domain file* lists predicates and action schemas; a *problem file* supplies objects, the initial state, and the goal. Planners such as **FF** and **Fast Downward** ingest PDDL and exploit the structured preconditions/effects for pruning and abstraction.

===== Key Points

* **STRIPS**: facts, preconditions, add/delete effects  
* **Forward planning**: initial → actions → goal  
* **Backward planning**: goal → reverse actions → initial  
* **Plan‑space search**: refine partial plans  
* **Frame problem**: what stays the same? → frame axiom  
* **PDDL**: domain + problem files, standardized syntax  
* **Planners**: FF, Fast Downward leverage structure for efficient search  

==== Technical Example

Write a PDDL domain for the classic **blocks‑world**:  

*Predicates*: `on(?x ?y)`, `clear(?x)`, `on-table(?x)`, `holding(?x)`, `hand-empty`.  
*Actions*: `pick-up`, `put-down`, `stack`, `unstack`, each with explicit preconditions and add/delete effects.

A corresponding problem file specifies the **initial configuration** (e.g., `on(A,B)`, `clear(A)`, `on-table(C)`) and the **goal configuration** (e.g., `on(C,B)`, `on(B,A)`).  

When a planner processes this PDDL, it **grounds** the action schemas by binding the object parameters, producing a concrete **successor function** that maps a state (a propositional encoding of facts) to its reachable successors. The search operates over an **implicit graph** generated on‑the‑fly by applying these grounded actions.

Lab 3 will first implement a generic successor function for state‑space search, then extend it to a **planner** that indexes actions and states, enabling queries such as “find a plan from state S₀ to goal G”. This demonstrates how the same search engine can support both document retrieval and action planning.

===== Key Points

* **PDDL blocks‑world**: predicates, actions, initial/goal  
* **Implicit graph**: generated by grounded actions  
* **Lab 3**: implement successor → extend to planner  
* **Action grounding**: bind parameters to objects, encode state as propositional facts  

==== Philosophical Reflection

The frame problem—*what stays the same when something changes?*—remains a central AI challenge. In the real world, actions have indirect consequences: moving a block may disturb dust, alter lighting, or shift the center of mass. Classical planning’s frame axiom (“nothing else changes”) works for toy domains but collapses under such complexity.

To handle richer environments we need **default persistence reasoning**: mechanisms that assume facts persist unless explicitly contradicted, and that can reason about indirect effects. This pushes us beyond pure STRIPS toward richer formalisms (e.g., situation calculus, temporal planning).

Nevertheless, within its scope, **planning as search** is powerful: it forces us to make the hidden combinatorial structure of a problem explicit, turning a vague “what should I do?” into a concrete search problem. As Eisenhower noted, *plans may be worthless when the world changes, but the act of planning clarifies goals and options*.

===== Key Points

* **Frame problem**: indirect effects challenge the frame axiom  
* **Limits of classical planning**: no continuous change, concurrency, exogenous events  
* **Default persistence reasoning** mitigates the frame problem  
* **Planning clarifies goals and options**
