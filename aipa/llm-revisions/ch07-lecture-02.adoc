=== 7.2: Vector Stores and Similarity Search
:leveloffset: 2

Imagine a personal assistant that can instantly pull up the exact paragraph from a ten‑million‑page legal archive that best answers a user’s question. How does it know *which* paragraph to fetch, and how quickly can it do so? 

[.epigraph]

*Vector stores: embed, index, retrieve by similarity.*

==== Conceptual Core

[.formal-definition]
****
*Formal Definitions (Vector Store, Embedding)*
A *vector store* is an index of vectors that supports similarity search (e.g., nearest neighbors). An *embedding* is a function that maps text (or other input) to a dense vector in ℝ^d such that semantically similar inputs map to nearby vectors. The vector store stores embeddings of documents; retrieval finds vectors nearest to the query embedding. See AIMA <<RussellNorvig2020>> (representation), <<Reimers2019>> (sentence embeddings).
****

Dense vectors represent meaning. Each document (or chunk) is embedded into a high‑dimensional vector—typically 384, 768, or 1536 dimensions—via a model trained to place semantically similar text near each other. Similarity is measured by cosine similarity or dot product. Given a query embedding, we find the *k* vectors closest to it. That is the core of vector search: embed, index, retrieve by similarity.

Exact nearest neighbor search is **O(n)**—we compare the query to every vector. For millions of documents, that is too slow. Approximate nearest neighbor (ANN) algorithms trade accuracy for speed. HNSW (hierarchical navigable small world) and IVF (inverted file index) are common index structures. They build a graph or partition of the space; search follows local structure and typically reaches a near‑optimal result in sublinear time. The trade‑off: we may miss the true nearest neighbor, but we usually get “good enough” results much faster. *In a 1 M‑doc corpus, exact search returns the true nearest neighbor 99.8 % of the time, while HNSW returns it 97 % of the time but in ~10 ms instead of ~800 ms—an acceptable loss for most RAG pipelines.*

Chunking matters. Documents are split into chunks—typically 256–512 tokens with optional overlap—before embedding. Chunk size affects retrieval quality: too small, we lose context; too large, we dilute relevance. Semantic chunking (split at sentence or paragraph boundaries) often outperforms fixed‑size splits. Overlap (e.g., 50 tokens) can help preserve context across chunk boundaries. The indexing pipeline: document → chunks → embed (via LLM or embedder) → index. Query: embed query → search index → return top‑k.

Retrieval as memory access: what gets retrieved shapes what the agent “knows.” The politics of similarity: embeddings encode the training distribution. Dominant perspectives, common phrasing, and majority viewpoints tend to cluster; minority or marginalized content may be under‑represented. Bias in embeddings becomes bias in retrieval—we retrieve what is “similar” in embedding space, which may not match what we would call relevant by other criteria.

===== Key Points

* Dense vectors: semantic representation in high dimensions
* Cosine similarity, dot product for nearest neighbors
* ANN: HNSW, IVF—trade accuracy for speed
* Chunking: size, overlap, semantic vs. fixed
* Pipeline: document → chunks → embed → index
* Query: embed → search → top‑k
* Retrieval = memory access; shapes what the agent knows
* Politics of similarity: bias in embeddings → bias in retrieval

==== Technical Walk‑through & Lab Prep

Index a document collection. Chunk documents—e.g., 512 tokens per chunk with 50‑token overlap. Embed each chunk using `llm.embed` or a dedicated embedder. Index in Chroma, FAISS, or another backend. Implement the search API: given a query (text or embedding), return the top‑k most similar chunks.

Compare exact vs. approximate search. Exact search scans all vectors; it is correct but slow at scale. ANN search uses the index structure; it is fast but may return a slightly different (often still relevant) set. For most applications, ANN is sufficient. Measure: recall@k (did we retrieve the “right” chunks?) and latency. Tune chunk size and overlap; different corpora benefit from different strategies.

This API becomes the interface for the RAG system in Lab 2, which will call `search` to retrieve context before generation. Document the API, chunking strategy, and index configuration according to the Lab 1 specification.

==== Philosophical Reflection

Retrieval as memory access: we do not retrieve “the” relevant document. We retrieve what is *similar* in embedding space. Similarity is not identity. Two chunks can be similar without being equivalent; the model receives an approximation of relevance. That approximation is often good enough—but it can fail. Queries that rely on rare phrasings, niche domains, or non‑standard terminology may retrieve poorly. The embedding model's biases become the retrieval system's biases.

The politics of similarity: who decides what counts as “similar”? Embeddings encode the training data. If the training data over‑represents certain voices, topics, or formats, retrieval will favor them. Minority viewpoints, dialectal variation, and domain‑specific jargon may be under‑represented. When we build memory systems, we are making epistemic choices—what gets stored, how it is chunked, how it is indexed. Retrieval is not neutral; it reflects those choices.

**Mitigation tip:** Augment the vector store with under‑represented sources and, where possible, apply debiasing transformations (e.g., projection‑based removal of gender or racial subspaces) to the embeddings before indexing. Periodically audit recall across demographic slices to surface systematic gaps.

Thus, the design of a vector store determines not only how fast an agent can recall, but also *what* it remembers and whose voices are amplified.

===== Key Points

* Similarity ≠ identity
* Retrieval = approximation of relevance
* Bias in embeddings → bias in retrieval
* Politics of similarity: who decides what counts?
* Similarity is often enough—but not always
* Memory = curated, approximate, politicized
* Mitigation: augment, debias, audit
