==== Conceptual Core

The simulator lets you run governance before you deploy. Instead of discovering policy gaps in production, you define policies, run scenarios (cost, fairness, epistemic impact), and integrate the governance tool into the agent so it can check itself. This lecture recaps the chapter, ties the governance tool to the capstone, and frames governed deployment as building governance in—not bolting it on. Ch12 will deploy the full stack; here we complete the institutional layer.

*Recap:* Chapter 11 has covered policies and constraints, cost control and rate limiting, responsible AI (fairness, bias, transparency), experimental rigor (A/B testing), GPU-backed evaluation (latency, cost, quantization), and epistemology and infrastructure. The *governance tool*—define_policy, check_compliance, simulate, report—embodies these in a single interface the agent (and operator) can invoke. It belongs in student-ai/ alongside memory, reasoning, and the orchestrator. The capstone does not "add" governance at the end; it deploys with governance integrated from the start.

*The governed agent*: governance is not external—it is built in. The agent calls check_compliance before sensitive actions; it can run simulate to test scenarios; it can report. From tool to institution: the governance tool encodes institutional requirements (policies, limits, epistemic and fairness considerations). The agent operates within a structure of accountability. To govern is to anticipate (Foucault, <<Foucault1980>>): we define policies for scenarios we foresee, and the simulator lets us test those scenarios before deployment. Governance is future-oriented—and the capstone is the place where it becomes concrete.

==== Technical Example

End-to-end: the agent uses the governance tool for compliance checks. Before sensitive actions (e.g. paid API call, user data write), it calls check_compliance; if blocked, it does not proceed or escalates. Run institutional scenarios in the simulator: e.g. audit (who did what), oversight (did we stay within policy?), redress (what if a user disputes?). The simulator reports outcomes—violations, cost, latency—so you can tune policies and document assumptions.

The deliverable is a governed system, not a system with governance added later.

==== Philosophical Reflection

The governed agent is one that has governance built in—we do not bolt it on after the fact. The governance tool encodes institutional norms: what is allowed, what is capped, what must be checked. The agent operates within that structure; it can refuse to act when policy forbids it. Ch12 deploys this: the capstone is a governed system. Evidence-based governance means we use the simulator to test scenarios and document what we expect—latency, cost, fairness, epistemic impact—before we deploy. That is what "governed deployment" means: not compliance as an afterthought, but governance as part of the design.

===== Key Points

* Governance built in; not compliance as afterthought
* Governance tool encodes institutional norms; agent operates within structure
* Evidence-based: simulator tests scenarios, document expectations before deploy
