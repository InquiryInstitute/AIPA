==== Conceptual Core

"Is this model fair?" sounds like one question—but it is many. Fairness has no single technical definition; different stakeholders mean different things, and the definitions can conflict. So we start from that tension: responsible AI requires us to make fairness and bias explicit, even when we cannot reduce them to one metric.

Fairness has multiple formal definitions. Demographic parity: equal approval (or positive) rates across groups. Equalized odds: equal true positive rate and false positive rate across groups. Calibration: predicted probabilities match observed frequencies. No single definition captures "fairness" in all contexts—fairness is contested. Different stakeholders prefer different definitions; they can be mutually incompatible. Bias can arise at multiple stages: data bias (training distribution reflects historical inequalities), model bias (the learned function amplifies or encodes bias), deployment bias (context of use introduces disparity). Audit for bias: compare metrics (accuracy, TPR, FPR, etc.) across demographic or other groups. Transparency includes explainability (why did the model produce this output?) and documentation (intended use, limitations, known failure modes). Tradeoffs: improving fairness (e.g., via reweighting or post-processing) may reduce overall accuracy. Fairness is contested—which definition should we optimize? Transparency as accountability: we document to enable scrutiny by users, regulators, and affected communities.

===== Key Points

* Fairness: parity, equalized odds
* Bias: data, model, deployment
* Transparency: explainability, documentation
* Tradeoffs: fairness vs. accuracy
* Fairness contested

==== Technical Example

Audit a model for bias. Split test data by group (e.g., demographic, geographic). Compare metrics: accuracy, TPR, FPR, precision, recall. Report disparities—e.g., "Group A: 90% accuracy; Group B: 75% accuracy." Document limitations: "Not validated for use case X." Write an intended use statement: what the model is designed for, and what it is not. Lab 2: Add bias audit to the governance simulator. Simulate scenarios: e.g., a model trained on biased data, or deployed in a context where one group is underrepresented. The simulator runs bias checks and produces reports for governance review. ==== Philosophical Reflection

Fairness is contested—there is no single correct definition. Different stakeholders (users, regulators, affected communities, developers) may prefer different definitions, and they can conflict. Transparency as accountability: we expose the system to scrutiny. Documenting limitations, intended use, and audit results enables others to assess whether the system is appropriate. The politics of fairness: who defines it? Who benefits from a given definition? Technical choices (which metric to optimize) have political consequences. The instructor note above reminds us to frame these as design choices with tradeoffs, not as problems with a single correct answer.

Bias is not only in the model—it can be in the data (historical inequity), the model (amplification), or the deployment context (who is underrepresented in use). The governance simulator's bias audit (Lab 2) lets you run scenarios and compare metrics across groups; the audit tool (Ch2) connects traceability so that when a decision harms someone, we can ask what the system knew and how it was trained.
